  - **新旧数据兼容**：进行中报警、救援中群聊、有效套餐、历史轨迹等历史数据与新逻辑匹配验证。

### 4.4 个人学习计划推进（《2026 年学习计划》第一阶段）

> 以下为个人学习计划落地内容，**非公司项目交付**，作为 AI 先锋赛参赛产品与下半年自动化闭环建设的基座。

  - 技术栈 `Python + pytest + requests + YAML + Allure`，目录含 `api_test_framework`（client/runner/assertions/schema/data/config）、`common`（allure 断言、验证码、yaml、日志、协议类型）、`testcases`、`yaml` 数据驱动；

- **技能收获**：掌握 pytest fixture/参数化、YAML 数据驱动、Allure 报告、Schema 断言分层。
- **关键技术探索 —— mem0 Bugfix 记忆闭环（已搭建技能包，验证中）**：
  - **做了什么**：把每次接口测试排障的「场景-根因-排查路径-修复方案-反模式」沉淀为结构化记忆，下次同类报错先召回复用，形成「报错→召回→复用/排查→沉淀」闭环。已落地 3 个 Cursor Rules 技能文件（schema 模板 + recorder 写入 + recall 召回）+ 接入官方 mem0 云端 MCP，并适配本项目技术栈。
  - **为什么用 mem0 而非通用 RAG**：mem0 专为"经验沉淀"设计，相比通用向量库多了 **`error_signature` 去重主键**（同问题 update 不重复 add）和**反模式记录**（"不要怎么修"），对修复场景针对性更强。
  - **下一步**：闭环验证 + 与禅道 bug ID 联动 + 团队共享。
- **当前阶段**：框架基座 + 文档自动生成原型已验证；mem0 Bugfix 记忆技能包已搭建并验证中；接口变更监控 + 智能回归选择待下半年攻坚。

## 六、反思与规划

### 6.1 半年反思

> 立足测试本职，结合上半年项目实践做反思。原则：点明问题与方向，不展开技术细节。

**做得比较扎实的地方**：

1. **核心项目缺陷闭环质量较高**：四大项目上半年 651 个缺陷、整体关闭率 91.2%，监控平台国际化与星联应急 1 期两个收尾项目关闭率均在 94% 以上。
2. **AI 已真正融入日常测试流程**：从用例生成、缺陷录入到报告输出，形成了「AI 干重复活、人来把关」的稳定工作模式，AI 先锋赛四等奖是对这条路的认可。
3. **复杂业务场景的测试能力有提升**：计费、状态机、多级权限、国际化等复杂场景，积累了成体系的测试思路。

**认识到需要改进的方向**：

1. **测试左移做得还不够**：上半年多个项目暴露「评审时对业务规则理解不够深、主要风险识别后移」，下半年争取在需求评审阶段就前置输出测试疑问与风险初判，而不是等测试设计阶段才补。
2. **回归测试仍以手工为主**：版本迭代密集时重复执行量大，是下半年自动化要解决的核心问题。
3. **非功能测试（性能/安全）仍是短板**：上半年精力聚焦功能交付与 AI 落地，这块推进有限，下半年逐步补齐。
4. **AI 辅助测试设计的稳定性有待提升**：复杂业务规则下 AI 生成的测试点偏通用化，需持续优化提示词与知识沉淀。

> 以上改进方向已在「6.2 下半年规划」中对应安排，此处不展开细节。

## 七、建议与思考

> 基于上半年项目实践与日常感受，从制度、流程、业务、资源四个维度提出建议，供领导参考。

### 7.1 公司 / 部门 / 制度建议

1. **薪酬福利相关信息的透明化**。部分福利项的计算规则与等级标准目前不够清晰，员工需要主动询问行政才能了解，建议在工资单或员工手册中明确：
   - **公积金缴纳基数**的计算方式与构成目前较为模糊，希望明确公式（基数如何确定、是否含各项补贴、调整周期等），让员工清楚自己的缴纳明细；
   - **交通补贴、餐补**的等级划分标准不够明确，建议随工资单注明所属档位与对应金额，避免每次都要单独询问。
   - 这类信息透明化能减少重复沟通成本，也增强员工对薪酬体系的信任感。
2. **多项目并行的工时与节奏管理**：下半年将并行 3 个新项目（监控四期图传、星联 3 期分销、SaaS 2 期）+ 遗留缺陷跟进，多项目切换对单人测试压力较大。建议部门层面建立项目测试工时可视化与负载预警机制，在排期阶段就识别资源瓶颈，避免测试成为交付链路的被动瓶颈。

### 7.2 工作流程优化建议（基于上半年项目体验）

3. **提测准入与节奏对齐**：上半年个别项目出现过「约定提测时间未按时提测、测试被动等待」的情况（如某项目核心链路测试因提测推迟延期约一周）。建议研发与测试在排期阶段约定**提测准入清单**（提测包可用、P0 自测通过、变更范围说明），降低测试空等与节奏被动。
4. **需求变更的同步与消化窗口**：上半年遇到过计费类规则在测试设计阶段才发生变更（如报位扣费口径调整），测试用例需返工重写。建议涉及计费、权限、状态机等核心规则的需求变更，**同步给测试并预留消化窗口**，避免变更与测试执行撞期。
5. **重构/大版本项目的变更清单交付**：重构类项目（如 SaaS 1 期）初期缺乏完整变更清单，测试需按业务链路倒推覆盖范围，效率偏低。建议研发在重构/大版本提测时**同步交付变更清单**（接口、缓存、分表、异步任务、权限边界），提升测试覆盖的精准度与效率。
6. **规则密集型需求的前置评审**：计费、分润、权限、状态机这类规则密集场景，上半年暴露过「规则未前置冻结、测试阶段反复发现同源问题」的痛点（如 2 期出现退款未扣减、余额负值等资金安全类缺陷）。建议这类需求在评审阶段就**冻结规则基线**（含统计口径、状态边界、红线），降低后期返工。

### 7.3 业务 / 产品发展建议（基于测试发现）

7. **业务强绑定内容的测试覆盖意识**：以监控平台国际化为例，初期的翻译问题不止在页面文案，还广泛存在于 Excel 导入导出、报警内容、WebSocket 推送异常、通信状态等**业务强绑定字段**，这类问题易漏测。建议后续涉及多端/多链路的项目，在需求阶段就梳理「全链路影响点清单」，作为测试覆盖依据。
8. **软硬件结合产品的测试资源保障**：公司核心产品多为软硬件结合，部分卫星/协议类终端实体设备不足时，测试只能靠模拟数据，置信度打折。建议核心产品在立项时**配套测试设备预算**（实体终端、多机型），保障关键链路的真实验证。
9. **计费/资金安全类逻辑的质量门禁**：上半年星联 2 期暴露的资金安全类缺陷（重复扣费、负值、未封禁）提示，涉及扣费、退款、余额、封禁的逻辑风险较高。建议这类逻辑在上线前增设**专项质量门禁**（账本核对、幂等与并发验证），守住资金安全红线。

### 7.4 资源与支持需求（下半年新项目所需）

10. **新项目并行期的测试资源补充**：下半年 3 个新项目并行，建议在测试设备（多机型安卓机、图传/卫星类实体终端）与协作工具上给予支持，保障并行交付质量。
11. **图传/分销等新业务域的学习与对齐支持**：钒星图传、星联 3 期分销涉及新业务领域（图传协议、分销分润模型），希望有与产品/业务方前置对齐需求的机会，降低测试设计阶段的理解偏差。
12. **AI 工具账号统一采购**：上半年已汇总团队所需 AI 工具清单，统一采购可提升落地一致性，也便于沉淀团队级 skills 服务于多项目并行。
