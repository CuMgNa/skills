- **主要工作内容及过程**：

  - 完成前端时区适配专项，梳理常见时区差对照表及 Sensors 国家/地区参数映射；

#### 项目三：【磐钴】星地多网融合指挥调度 SaaS 平台 1 期（监控平台重构）

- **项目背景与目标**：以 pg-podium-monitor 核心监控服务为基础，对监控平台终端定位、轨迹、消息通信、SOS 报警、求救群聊、电子围栏、账号权限、套餐扣费、WebSocket 推送、第三方对接等核心能力进行**重构验证**，确保"原有业务能力不丢失、核心链路行为一致、异常场景可控"。
- **个人角色与具体职责**：核心测试工程师（独立编制《重构测试方案》V1.0，2026-05-06）。
- **重构技术重点**：MongoDB 分表（base/loc/file/alarm 多数据源）、Redis 缓存、WebSocket 实时推送、异步队列、一级/二级/三级账号权限边界。
- **主要工作内容及过程**：
  - 独立编制《重构测试方案》V1.0，定义 **12 个核心模块**测试范围、准入/准出标准、8 类测试类型、P0~P3 分层、风险评估与里程碑；
  - 完成 12 大核心模块测试大纲（登录认证、设备与分组管理、用户与账号体系、终端定位与轨迹、消息通信、SOS 报警与报平安、求救群聊、套餐与扣费、电子围栏、WebSocket 实时推送、第三方对接、订阅与通知推送）；
  - 完成主链路冒烟（8 条核心链路打通）+ 一轮/二轮功能测试 + 线上环境测试；Q2 完成 1 期验收报告与项目验收总结。
- **重构测试核心原则**（重构场景特有）：①先数据正确再关注展示（每条关键链路结合数据库/缓存/接口日志三层验证）；②每个扣费动作必验幂等；③每个权限动作必验三级账号边界；④新旧数据兼容性逐项核查。

### 4.4 个人学习计划推进（《2026 年学习计划》第一季度）

- **学习方向**：第一阶段（3—6 月）「AI 在测试中的应用探索」，核心目标为实现 AI 驱动的接口自动化测试闭环。
- **落地产物 —— `jkpt_api_test` 接口自动化框架**：
  - 技术栈 `Python + pytest + requests + YAML + Allure`；
  - 框架由数据驱动层、HTTP 客户端、断言引擎、协议编解码、测试脚本等模块组成，覆盖监控平台 9 大核心模块接口；上半年持续迭代；
  - 已覆盖 login、group_controller、terminal、alarm、alarm_settings、batch_terminal、field_template、location、bd_protocol 等监控平台核心接口；
  - 通过 **Cursor MCP 连接 Apifox**，实现基于接口定义自动生成监控平台接口自动化脚本；Allure 可视化报告已集成。
- **技能收获**：掌握 pytest fixture/参数化、YAML 数据驱动、Allure 报告、Schema 断言分层、纯 Python 协议编解码。
- **关键技术探索 —— mem0 Bugfix 记忆闭环**：
  - **做了什么**：把每次接口测试排障的「场景-根因-排查路径-修复方案-反模式」沉淀为结构化记忆，下次同类报错先召回复用，形成「报错→召回→复用/排查→沉淀」闭环。
  - **当前状态（如实）**：已独立实现 mem0 MCP 服务（`tools/mem0-mcp/server.py`，用 FastMCP 封装 `add_memory`/`search_memory` 工具，走 mem0 云端 API）+ 3 个 Cursor Rules 技能文件（schema 模板 + recorder 写入 + recall 召回）；git 有 2026-05-08、06-02 两次 mem0 提交。**已实现 MCP 服务与技能包，但尚未接入主测试框架的运行时自动调用流程**，闭环命中率验证 + 禅道 bug ID 联动 + 接口变更监控待下半年推进。
  - **为什么用 mem0 而非通用 RAG**：mem0 专为"经验沉淀"设计，相比通用向量库多了 `error_signature` 去重主键和反模式记录，对修复场景针对性更强。

## 七、建议与思考


### 7.1 公司 / 部门 / 制度建议

1. **多项目并行的工时与节奏管理**：下半年将并行多个新项目 + 遗留缺陷跟进，多项目切换对单人测试压力较大。建议部门层面建立项目测试工时可视化与负载预警机制，在排期阶段就识别资源瓶颈。
2. **薪酬福利信息透明化**：公积金缴纳基数、交通补贴/餐补的等级划分标准目前不够清晰，建议在工资单或员工手册中明确，减少重复沟通成本。

### 7.2 工作流程优化建议

1. **提测准入与节奏对齐**：上半年个别项目出现过"约定提测时间未按时提测、测试被动等待"的情况。建议研发与测试在排期阶段约定**提测准入清单**（提测包可用、P0 自测通过、变更范围说明）。
2. **需求变更的同步与消化窗口**：上半年遇到过计费类规则在测试设计阶段才发生变更（如报位扣费口径调整），测试用例需返工重写。建议涉及计费、权限、状态机等核心规则的需求变更，**同步给测试并预留消化窗口**。
3. **重构/大版本项目的变更清单交付**：重构类项目（如 SaaS 1 期）初期缺乏完整变更清单，测试需按业务链路倒推覆盖范围。建议研发在重构/大版本提测时**同步交付变更清单**。


### 7.3 业务 / 产品发展建议

1. **原始协议数据的提供**：部分协议从未使用真实设备测试，测试侧不清楚其上报的原始数据格式。希望研发/业务方提供各协议的真实上报报文样本。
2. **移动端 UI 兼容性系统覆盖**：APP/小程序当前以功能验证为主，UI 兼容性几乎不系统覆盖。建议基于客户机型分布建立机型覆盖矩阵，引入云真机平台覆盖主流机型回归。

### 7.4 资源与支持需求

1. **真机与协议验证的专项支持**：下半年新项目涉及图传/卫星/通信等协议，建议在实体终端设备、协议录制与回放工具、云真机平台账号上给予专项支持。
2. **主流模型中转站统一搭建**：希望公司统一搭建主流大模型的中转服务，团队成员共享使用，避免各自散购账号、成本分散且版本不一致的问题，同时便于沉淀团队级 AI 使用规范，提升整体落地效率。
