# 02 对讲群添加成员（邀请）逻辑点梳理

**适用范围**：仅覆盖「对讲群」邀请设备入群链路（入口 → 设备列表过滤 → 邀请权限 → 扣费 → 邀请处理 → 换群 → 并发 → 退还）。**邀请对象仅限天通应急救援棒（TT_RESCUE_STICK）设备**。

## 1. 角色与权限规则（谁能添加成员）
| 角色 | 能否添加成员 | 说明 |
|---|---|---|
| **群主**（个人 / 一级 / 二级 / 三级账号均可） | ✅ 可邀请 | 仅对**自己创建**的群；星豆充足创建对讲群者自动成为群主 |

对讲群里除群主外的成员，本身就是**被邀请入群的终端（设备）**，不存在「添加成员」这个动作；邀请动作始终由**群主**发起。
---
## 2. 设备类型与设备列表过滤规则（多账号 / 多标签核心）
### 2.1 设备类型限制
邀请设备列表**只显示天通应急救援棒（TT_RESCUE_STICK）**类型设备。

### 2.2 设备标签来源回顾
- 标签取值：**我的 / 用户 / 好友 / null**，优先级 **我的 > 用户 > 好友**。
- 形成路径：扫码绑定 → **我的**；个人授权派生 → **用户**（仅企业账号会出现）；扫码关注 → **好友**。
- 一台设备最多被 **1 个个人账号 + 1 个一级企业账号** 同时绑定（双绑场景，均可为「我的」）。

### 2.2.1 设备-账号关系是「多维」的
标签（我的/用户/好友）是「**账号 × 设备**」的**相对属性**，不是设备的全局属性。
| 关系维度 | 形成方式 | 可持有的账号 | 数量约束 | 该账号看到的标签 |
|---|---|---|---|---|
| 个人「我的」 | 个人扫码绑定 | 个人账号 | **最多 1 个** | 我的 |
| 企业「我的」 | 一级企业扫码绑定（ENTERPRISE） | 一级企业账号 | **最多 1 个** | 我的 |
| 企业「用户」 | 个人将设备授权派生给企业（FOLLOW） | 企业账号 | **可多个** | 用户 |
| 「好友」 | 扫码关注 | 个人 / 企业账号 | **可多个** | 好友 |

同一账号若同时具备多种关系，按优先级 **我的 > 用户 > 好友** 取最高标签展示。

### 2.3 设备列表过滤规则（按邀请方）
- **个人邀请方** —— 看设备是否已被**某个个人账号**关联为「我的」（即存在个人「我的」归属）。
- **企业邀请方** —— 仅看设备是否被**本企业**关联为「我的」（ENTERPRISE 绑定）；「用户」（FOLLOW 授权派生）与「好友」（关注）的设备不展示。

**个人账号进入邀请列表的两个前提**：①该设备已在账号A的设备管理中；②该设备已被某个个人账号绑定为「我的」。

**企业账号判定核心**：仅「我的」（ENTERPRISE 绑定）的设备可邀请；「用户」与「好友」的设备不展示。

### 2.3.3 已在群 / 邀请中状态过滤（重复邀请拦截）
只要目标设备**已在本群**、或**已有指向本群的待处理邀请通知**，按入口区分处理、**均不扣费**：**批量添加** → 列表**直接过滤不展示**；**扫码添加** → 现场**提示拦截**。计费只发生在「该设备此前既不在本群、也无本群 pending」一种有效情形（扣 1 笔）。

## 4. 添加成员主流程（US-群-04）
- **入口**：群聊页底部「添加成员」（仅群主可见）或群信息页入口。
- **前置**：当前用户为群主；群内成员数未达上限。
- **关键校验顺序**：成员上限 → 余额 → 设备归属 → 是否已在他群。
- 流程：群主点击添加 → 校验成员上限(满则提示不扣费) → 选设备(手填卡号/扫码SN) → 余额校验 → 设备归属判断(自己设备/他人设备) → 是否在其他活跃群(自己设备直接退原群入新群/他人设备发邀请通知) → 扣除星豆

### 全局前置校验（P0–P3）
- P0 角色：必须是群主本人
- P1 群容量：成员数未达上限（个人 5 / 企业 100，可配置）
- P2 账号状态：群主账号冻结 / 过期 → 跳转登录页阻断
- P3 设备类型：仅天通救援棒（TT_RESCUE_STICK）可入对讲群

## 5. 星豆消耗与退还（计费 / 退还）
### 5.1 计费参数（个人 / 企业统一）
- **唯一参数控制**：邀请扣费由管理后台「**邀请成员扣除星豆**」单一参数（开关 + 数值，默认开启、20 豆）控制。
- **个人与企业完全一致**：都走同一个后台参数，不存在按账号类型分别定价。
- **扣费主体**：扣**发起操作账号自身**星豆 —— 个人扣个人；子账号操作扣子账号（子账号星豆独立、不共享、不扣一级）。
- **扣费时点**：**邀请即扣**。
- **快照固化**：邀请即扣时落一条星豆消耗快照流水（消耗金额、扣费账号、退还开关状态、过期时长）；后台中途改配置不影响在途邀请。
- **退还按快照回放，不重算**。

### 5.4 可退还场景（统一「按退还开关」快照判定）
**退还总口径**：**无「无条件退还」一档**，是否退 = **发起时快照的「退还开关」**——开启则原路退回扣费账号，关闭则不退；退还金额一律 = 该笔消耗快照流水的原额；**同一笔邀请星豆只退一次**。
| # | 可退还场景 | 类型 |
|---|---|---|
| 1 | 被邀请人主动拒绝 | 异步失效 |
| 2 | 超时未处理失效 | 异步失效 |
| 3 | 待处理期间群已结束 | 异步失效 |
| 4 | 待处理期间群已满员 | 异步失效 |
| 5 | 并发失败（多群抢同设备） | 异步失效 |
| 6 | 设备已解绑 | 动态异常 |
| 7 | 设备归属变更 | 动态异常 |
| 8 | 账号已注销 | 动态异常 |
| 9 | 设备类型已变更 | 动态异常 |
| 10 | 批量即时阶段失败回滚 | 即时强一致 |

### 5.5 不可退还场景
| # | 不退场景 | 说明 |
|---|---|---|
| 1 | 入群成功后被群主移除成员 | 服务已兑现，不退 |
| 2 | 换群退出原群 | 退出旧群那笔不退（新群按新邀请另扣） |
| 3 | 已成功入群的正常消耗 | 自己设备直接入群 / 他人同意入群 |
| 4 | 群正常解散 / 结束后已入群成员的已扣星豆 | 与「待处理期间群结束→失效退」区分 |

### 5.6 关键边界
- **只退一次**：多原因同时命中也只退一笔（快照流水带「已退还」标记保证幂等）。
- **原路退回**到当初扣费账号，金额 = 快照原额。
- **不在邀请记录模块展示扣费 / 退还**，需到「**星豆记录 / 明细**」查看。
- **扣费开关关闭** → 邀请免费、退还开关自动置灰禁用。
- 退还在后台**不计入「退款」统计**（退款口径仅含套餐）。
---
## 6. 关键规则总结
1. **权限**：仅群主可邀请。
2. **设备范围与过滤**：仅天通救援棒；个人看「有个人『我的』归属」的设备，企业仅看本企业 ENTERPRISE 绑定设备，两条链路按归属严格隔离。
3. **入群与唯一性**：自己设备直接入群、他人设备发通知待确认；一台设备同时仅属一个活跃群，并发仅一个成功，接受新邀请自动退原群。
4. **扣费 / 退还**：个人 / 企业同一参数，默认 20 豆、邀请即扣；失效按退还开关退、批量即时失败整批回滚，移除 / 换群退出 / 已入群不退，只退一次。
