# 05 天通应急救援棒终端上报消息路由转发逻辑

本文梳理**天通应急救援棒（TT_RESCUE_STICK）**「终端 → 平台」方向上报消息的路由转发逻辑，覆盖语音与四类报位，及 SOS 求救群聊 / 对讲群 / 普通聊天室三个会话载体的转发规则。

## 一、适用范围
- 仅适用于**天通应急救援棒（TT_RESCUE_STICK）**设备的「终端 → 平台」上报方向。
- 其他类型设备沿用旧逻辑（仅普通通信，可收文本/语音/图片/报警/报平安），不在本逻辑内。
- 涉及三个会话载体：**SOS 求救群聊**、**对讲群（PTT 群）**、**普通聊天室（平台单通信）**。
---
## 二、消息类型总览（1 类语音 + 4 类报位）
| 协议 | 标识 | 类型 | 非报警状态 | 报警状态下 |
|---|---|---|---|---|
| 0x02 | — | 语音（短音） | 语音气泡，按路由展示 | 同左 |
| 0x01 | 0 | 普通位置 | 仅写轨迹，无气泡 | 向 SOS 求救群聊推「已取消报警」文本 + 关闭 SOS 求救群聊（救援完成），同时写轨迹 |
| 0x01 | 1 | 按键 SOS | 派生报警文本，建/进 SOS 求救群聊 | 同左 |
| 0x01 | 2 | 落水 SOS | 派生报警文本，建/进 SOS 求救群聊 | 同左 |
| 0x01 | 10 | 取消 SOS | 仅状态变更，无气泡 | 向 SOS 求救群聊推「已取消报警」文本 + 关闭 SOS 求救群聊（救援完成），同时写轨迹 |

- **L1**：标识必须 ∈ {0,1,2,10}，否则记录日志并丢弃。
- **L2**：SOS（1/2）不是独立文本，是位置上报派生的报警文本；报警文本 + 位置点属同一事件。
- **L3**：「已取消报警」文本触发条件 = 设备当前处于报警状态 + 收到普通位置(0) 或 取消SOS(10)；非报警态下两者均不产生聊天文本。
---
## 三、总路由入口
- **L4**：平台收到消息 → 接收并存储 → 查询设备绑定的所有账号。
- **L5**：**未绑定任何账号（无我态）** → 仅存档，不转发、不展示。
- **L6**：**已绑定** → 按消息类型分流；账号路由采用**逐个关联账号判断**。
- **L7**：SOS 求救群聊与对讲群是**两个独立、可并存**的判断维度，命中谁就转发谁，互不排斥；普通聊天室为兜底。
---
## 四、语音消息路由（0x02）
逐账号判断，SOS 求救群聊与对讲群可同时命中。
语音路由结果矩阵：
| 在 SOS 求救群聊 | 在对讲群 | 转发目标 |
|---|---|---|
| 是 | 是 | **SOS 求救群聊 + 对讲群**（两者都发） |
| 是 | 否 | 仅 SOS 求救群聊 |
| 否 | 是 | 仅对讲群 |
| 否 | 否 | 普通聊天室 |

- **L8**：语音转发至对讲群时，按 PTT 下行广播给群内其他终端。
- **L9**：救援完成（SOS 求救群聊已关闭）后再上报的语音 → 进普通聊天室。
---
## 五、位置路由总流程（标识 0 / 1 / 2 / 10）
### 5.1 SOS 报警（标识 1 / 2）
- **L10**：先判断有无账号关联设备；无 → 仅存档。
- **L11**：有账号 → 判断有无已存在的 SOS 求救群聊：**有**则不重复创建；**无**则创建 SOS 求救群聊 + 更新定位 + 推送报警文本。
- **L12**：随后判断终端是否在对讲群：**在** → 对讲群镜像该报警文本；**不在** → 结束。
- **L13**：同一条 SOS 报位即使同时进 SOS 求救群聊与对讲群，**发送设备只扣 1 次报位**。
- **L14**：报警文本由位置派生，不额外扣短音/报位。

### 5.2 普通位置（标识 0）/ 取消报警（标识 10）
- **L15**：先判断有无账号关联设备；无 → 仅存档。
- **L16**：有账号 → 判断有无已存在的 SOS 求救群聊：**无** → 仅更新定位；**有** → 更新定位 + 向 SOS 求救群聊推一条派生文本「设备已取消报警」，并将群置为「救援完成」。
- **L17**：普通位置 / 取消报警的「设备已取消报警」文本**只进 SOS 求救群聊，不推送对讲群**；普通报位也不产生对讲群报位气泡、不扣对讲群额度。
- **L18**：以上位置点仍照常写入轨迹，并参与报位计费。
