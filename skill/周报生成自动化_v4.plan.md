# Weekly Report — 周报生成（v4）

## 目标定位

- 用户说"生成周报""写周报""本周总结""周工作日志"等意图时触发。
- 动态拉取所有进行中项目（`isArchived=false AND isSuspended=false`），数量不写死、支持翻页。
- **执行人 ID 与项目自定义字段 ID 运行时动态解析**，不写死（见「第零步：配置解析」）。
- 周报写入 Notion，遵循 Notion 模板的**结构与填写原则**（非示例正文），页面命名：`软件研发周报 YYYY-MM-DD · 童美娜`（日期取本周一）。
- 严格区分三个时间维度：**本周已完成 / 本周进行中 / 下周计划**。

## 配置（语义锚点）

```yaml
# —— 动态解析（第一类/第二类，运行时获取，不写死）——
identity:
  mode: current_user        # 默认：取当前登录者；可改 byName: 童美娜
  overrideExecutorId: ~     # 仅当自动解析失败时手动填
fieldHints:                 # 按“名称/类型”解析，逐项目动态映射
  实际工时: { type: worktime, prefer: "worktime:total:worktime" }
  计划工时: { type: worktime, prefer: "worktime:total:plantime" }
  最新评论: { nameContains: 评论 }

# —— 固定值（第三类，Notion 文档不会变更，直接写死）——
notion:
  templatePageId: "3615667c6d3a80089c1aeceaebf33711"   # 软件研发周报模板
  targetParentId: "3615667c6d3a80de9cc1fdb63d3424d4"   # 周报目标父页面
```

> 设计原则：**语义锚点（姓名/字段名/类型）写进 skill，易变 ID 运行时解析**。Notion 页面因为稳定不变，作为唯一的硬编码例外。

## 输入

1. **时间范围**（可选，默认本周一 00:00 至本周日 23:59，Asia/Shanghai）：`weekStart / weekEnd / nextWeekStart / nextWeekEnd`。
2. **执行人**：默认取当前登录者（见配置 `identity`），不再手填 ID。
3. **是否纳入参与者任务**（可选，默认否）。

> 所有 Teambition 时间字段为 UTC，比较与展示前统一 +8h 转北京时间。

## MCP 调用前置

调用任何 MCP 工具前，先读取该工具的 schema 确认参数结构与字段名，不要凭记忆传参。

### Teambition MCP（查任务）

| 工具 | 用途 |
|------|------|
| `获取本人用户信息` | 解析执行人 ID（第一类动态解析），返回字段含 `userId`（唯一标识）|
| 项目自定义字段定义类工具 | 解析每个项目的 customFields 定义（第二类动态解析）|
| `get_user_projects` | 获取所有进行中项目（翻页直到取完）|
| `search_task_ids_by_tql` | 按项目、执行人、时间窗口筛选任务 ID |
| `get_task_detail` | 批量读取任务详情（每批 ≤25 个 ID）|

### Notion MCP（写周报）

| 工具 | 用途 |
|------|------|
| `API-post-search` / 查询 | 检测当周是否已存在同名周报（幂等）|
| `API-retrieve-page-markdown` | 读取模板结构（templatePageId）|
| `API-post-page` | 在 targetParentId 下创建周报页面 |
| `API-patch-block-children` | 分批写入内容块（每批 ≤100 个 block）|

## 执行流程

### 第零步：配置解析（动态解析执行人）

1. **执行人 ID**：
   - `identity.mode = current_user`：调用 `获取本人用户信息` 工具，取返回字段中的 `userId` 作为执行人 ID（周报默认”给自己生成”，执行人=调用者）。
   - `identity.mode = byName`：按姓名在成员中查找（`姓名 → memberId`）。
   - 两者都失败且 `overrideExecutorId` 为空：**fail fast**，提示无法确定执行人。
2. **时间窗口**：若未指定，按 Asia/Shanghai 计算本周一~本周日及下周一~下周日。
3. 输出 `executorId / weekStart / weekEnd / nextWeekStart / nextWeekEnd` 供后续使用。

### 第一步：拉取进行中项目（支持翻页）

```javascript
get_user_projects({ pageSize: "50", nextToken: <上一页token> })
// 循环翻页直到无 nextToken；过滤 isArchived=false 且 isSuspended=false；数量动态
```

### 第二步：逐项目解析自定义字段映射（动态解析 customFields）

> 关键：Teambition 自定义字段 ID 是**按项目定义**的，不同项目不一样。不能用一个 ID 套所有项目。

对每个进行中项目，读取其自定义字段定义，按 `fieldHints` **按名称/类型匹配**出真实字段 ID，构建逐项目映射表并缓存：

```
fieldMap[projectId] = {
  实际工时: <匹配 type=worktime 且名含“实际工时”的字段；优先用系统键 worktime:total:worktime>,
  计划工时: <同上，优先 worktime:total:plantime>,
  最新评论: <名称含“评论”的文本字段 ID>
}
```

解析规则：
- **工时优先用系统稳定键** `worktime:total:worktime` / `worktime:total:plantime`（跨项目稳定）；项目级 customField 工时字段仅作兜底。
- 某项目某字段**解析不到**：该字段在该项目留空并记告警，**绝不套用别的项目的字段 ID**（这正是写死最隐蔽的坑）。
- 映射表按 projectId 缓存，避免重复查询。

### 第三步：按时间窗口分类查任务

对每个进行中项目执行三组带时间条件的 TQL（`<executorId>` 为第零步解析结果）：

```
# A. 本周已完成
projectId = <项目ID> AND executorId = <executorId> AND isArchived = false
  AND isDone = true AND accomplishTime >= <weekStart> AND accomplishTime <= <weekEnd>

# B. 本周进行中
projectId = <项目ID> AND executorId = <executorId> AND isArchived = false
  AND isDone = false AND progress > 0 AND updated >= <weekStart> AND updated <= <weekEnd>

# C. 下周计划
projectId = <项目ID> AND executorId = <executorId> AND isArchived = false
  AND isDone = false AND dueDate >= <nextWeekStart> AND dueDate <= <nextWeekEnd>
```

> 若启用「纳入参与者」，将 `executorId = ...` 扩展为 `(executorId = ... OR involveMembers 包含 ...)`。字段可查询性以 schema 为准。

### 第四步：批量读取详情

`get_task_detail` 批量读取（每批最多 25 个 ID），字段取值**通过第二步 fieldMap 动态定位**，不再引用硬编码 ID：

| 字段 | 用途 |
|------|------|
| `content` | 任务名称 |
| `priority` | 优先级（2=非常紧急，1=紧急，0=普通，-10=较低）→ 排序/紧急标记 |
| `progress` | 进展百分比 |
| `isDone` | 完成状态 |
| `dueDate` | 截止日期（UTC +8h）|
| `accomplishTime` | 完成时间（UTC +8h）|
| `tags` / 标签 | 小任务按标签合并 |
| `customFields[ fieldMap[pid].最新评论 ]` | 最新评论（作说明/风险来源）|
| `worktime:total:worktime`（首选）/ `customFields[ fieldMap[pid].实际工时 ]`（兜底）| 实际工时 |
| `worktime:total:plantime`（首选）/ `customFields[ fieldMap[pid].计划工时 ]`（兜底）| 计划工时 |

> 评论嵌在 `customFields` 中，无需单独调用评论读取工具。工时取数：系统键优先 → 项目字段兜底 → 都缺则留空。

### 第五步：读取并提炼 Notion 模板结构

```
API-retrieve-page-markdown({ page_id: notion.templatePageId })
```

- **只提取结构骨架与填写原则**，剥离模板中所有“模板：”“示例：”占位正文。
- 保留三大板块：本周工作进展 / 风险与问题 / 下周工作计划；保留“写结果、写说明、可跟踪”原则。

### 第六步：内容组织规则

**本周工作进展**
- 项目类：按项目分组，`【模块/阶段/任务】 具体任务；说明：…；状态：已完成/进度N%`。
- 非项目类/零散小任务：按标签【日常运维】【需求支持】【质量保障】【协同支撑】【效率改进】**合并汇总**，写“总量 + 1~2 个典型案例”。
- “说明”优先取最新评论；无评论时用任务描述提炼，可注明“指标待上线/灰度后补充”。

**风险与问题**
- 来源：阻塞状态任务、评论含风险/阻塞关键词的任务。
- 格式：`【模块/任务】 现象：…；影响：…；需要：…；期望解决时间：…`。
- 若无法自动识别，保留板块标题并写“（本周无重大风险，或待人工补充）”，不整段省略。

**下周工作计划**
- 来源：第三步 C 组（dueDate 落在下周）。
- 格式：`【模块/任务】 具体任务；截止时间：<dueDate 北京时间>；完成标志：…`。
- “完成标志”优先从描述/评论提炼；无法提炼时留“待补充”。

### 第七步：写入前自检（对应模板“30 秒自检”）

- [ ] 每条进展是否写了“说明”而非仅“做了什么”？
- [ ] 风险项是否写清“需要谁支持 + 时间点”？
- [ ] 下周计划是否写清“截止时间 + 完成标志”？
- [ ] 是否已剥离模板示例正文？
- [ ] 小任务是否已按标签合并、无流水账？
- [ ] 是否有项目因字段解析失败而留空？（有则列出告警）

### 第八步：幂等检查 + 创建并写入周报页面（固定父页）

1. **幂等检查**：在 `notion.targetParentId` 下查是否已存在 `软件研发周报 <weekStart> · 童美娜`；已存在则提示选择“更新/另建副本”，默认不重复创建。
2. **创建页面**：
```javascript
API-post-page({
  parent: { page_id: notion.targetParentId },   // 固定：3615667c6d3a80de9cc1fdb63d3424d4
  properties: { title: [{ text: { content: "软件研发周报 <weekStart> · 童美娜" } }] },
  children: [...]
})
```
3. **分批写入**（每批 ≤100 个 block）。
4. **空结果跳过**：某项目本周对她无任务时跳过，不生成空标题段。

## 输出

成功：`notionPageId` / `notionUrl` / `executorId` / `weekStart` / `weekEnd` / `projectCount` / `taskStats`（已完成/进行中/下周计划计数）/ `fieldResolveWarnings`（字段解析告警列表）/ `reportStatus = success`。
失败：`reportStatus = failed` / `error` / `writtenPartial`。

## 失败策略

- 执行人无法解析且无 override：**fail fast**，不查任务。
- Teambition 项目/任务查询失败：终止，不写 Notion。
- 某项目某自定义字段解析失败：该字段留空 + 记告警，不套用其他项目 ID，不中断整体生成。
- 某字段 TQL 不可查询：降级为先拉任务再在 detail 阶段按时间过滤。
- Notion 读取模板失败：使用默认结构继续。
- Notion 写入失败：返回已写入部分和原因，供续写重试。

## 工时换算

- 毫秒 → 分钟：`÷ 60000`；毫秒 → 小时：`÷ 3600000`。
- 实际工时取数优先级：系统键 worktime:total:worktime（毫秒换算）> 项目字段兜底 > 留空。

## 变更记录（相对 v3）

1. **执行人 ID 动态解析**：默认调用 `获取本人用户信息` 取 `userId`，或按姓名查；不再写死 `050542485931269363`；解析失败 fail fast。
2. **customFields 动态解析**：逐项目读取字段定义，按名称/类型构建 `fieldMap` 并缓存；工时优先用系统稳定键；解析不到留空告警、不跨项目套 ID。
3. **Notion 页面 ID 保持固定**：模板页与目标父页直接写死于 `notion` 配置（文档不会变更）。
4. **`获取本人用户信息` 工具名明确化**：替换「类工具」模糊描述为正式工具名。
