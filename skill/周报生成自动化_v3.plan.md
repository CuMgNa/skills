# Weekly Report — 周报生成（v3）

## 目标定位

- 用户说"生成周报""写周报""本周总结""周工作日志"等意图时触发。
- 动态拉取所有进行中项目（`isArchived=false AND isSuspended=false`），数量不写死、支持翻页。
- 周报写入 Notion，遵循 Notion 模板的**结构与填写原则**（非示例正文），页面命名：`软件研发周报 YYYY-MM-DD · 童美娜`（日期取本周一）。
- 严格区分三个时间维度：**本周已完成 / 本周进行中 / 下周计划**，避免把历史在办任务全量混入。

## 输入

1. **时间范围**（可选，默认本周一 00:00 至本周日 23:59，时区 Asia/Shanghai）
   - `weekStart`：本周一日期，如 `2026-06-15`
   - `weekEnd`：本周日日期，如 `2026-06-21`
   - `nextWeekStart` / `nextWeekEnd`：下周一~下周日，用于「下周工作计划」筛选
2. **执行人 ID**：`050542485931269363`（童美娜）
3. **Notion 模板页面 ID**：`3615667c6d3a80089c1aeceaebf33711`
4. **Notion 目标父页面 ID**：`3615667c6d3a80de9cc1fdb63d3424d4`
5. **是否纳入参与者任务**（可选，默认否）：为真时同时匹配 `executorId` 与参与者字段。

> 所有 Teambition 时间字段为 UTC，比较与展示前统一 +8h 转北京时间。

## MCP 调用前置

调用任何 MCP 工具前，先读取该工具的 schema 确认参数结构与字段名，不要凭记忆传参。

### Teambition MCP（查任务）

| 工具 | 用途 |
|------|------|
| `get_user_projects` | 获取所有进行中项目（翻页直到取完） |
| `search_task_ids_by_tql` | 按项目、执行人、时间窗口筛选任务 ID |
| `get_task_detail` | 批量读取任务详情（每批 ≤25 个 ID） |

### Notion MCP（写周报）

| 工具 | 用途 |
|------|------|
| `API-post-search` / 查询 | 检测当周是否已存在同名周报（幂等） |
| `API-retrieve-page-markdown` | 读取模板结构 |
| `API-post-page` | 创建周报页面 |
| `API-patch-block-children` | 分批写入内容块（每批 ≤100 个 block） |

## 执行流程

### 第一步：确定时间窗口

- 若用户未指定，按 Asia/Shanghai 计算本周一~本周日，以及下周一~下周日。
- 输出 `weekStart / weekEnd / nextWeekStart / nextWeekEnd`，后续所有筛选都基于这些边界。

### 第二步：拉取进行中项目（支持翻页）

```javascript
get_user_projects({ pageSize: "50", nextToken: <上一页token> })
// 循环翻页直到无 nextToken
// 过滤：isArchived=false 且 isSuspended=false
// 数量动态，以实际返回为准
```

### 第三步：按时间窗口分类查任务（关键改动）

对每个进行中项目执行**三组**带时间条件的 TQL 查询，避免拉出全量历史在办任务：

```
# A. 本周已完成
projectId = <项目ID> AND executorId = 050542485931269363 AND isArchived = false
  AND isDone = true AND accomplishTime >= <weekStart> AND accomplishTime <= <weekEnd>

# B. 本周进行中（本周有进展/更新）
projectId = <项目ID> AND executorId = 050542485931269363 AND isArchived = false
  AND isDone = false AND progress > 0 AND updated >= <weekStart> AND updated <= <weekEnd>

# C. 下周计划（截止落在下周）
projectId = <项目ID> AND executorId = 050542485931269363 AND isArchived = false
  AND isDone = false AND dueDate >= <nextWeekStart> AND dueDate <= <nextWeekEnd>
```

> 说明：若启用「纳入参与者任务」，将 `executorId = ...` 扩展为 `(executorId = ... OR involveMembers 包含 ...)`。
> 实际 TQL 字段名以 schema 为准（如 `updated` / `accomplishTime` / `dueDate` 的可查询性需先验证，不可查询时改用 detail 阶段过滤）。

### 第四步：批量读取详情

`get_task_detail` 批量读取（每批最多 25 个 ID），读取字段：

| 字段 | 用途 |
|------|------|
| `content` | 任务名称 |
| `priority` | 优先级（2=非常紧急，1=紧急，0=普通，-10=较低）→ 用于排序与紧急标记 |
| `progress` | 进展百分比 |
| `isDone` | 完成状态 |
| `dueDate` | 截止日期（UTC，+8h 转北京时间）|
| `accomplishTime` | 完成时间（UTC，+8h）|
| `tags` / 标签 | 用于小任务按标签合并汇总 |
| `customFields` → `5ebdfef2bf8ac50eb6e149d1` | 最新评论内容（作说明/风险来源）|
| `customFields` → `5ec6001f852a6181e4028090` | 实际工时（分钟，**主**）|
| `customFields` → `worktime:total:worktime` | 实际工时（毫秒÷60000，**兜底**，当主字段缺失时使用）|
| `customFields` → `worktime:total:plantime` | 计划工时（毫秒÷60000 转分钟）|

> 注：评论嵌在 `customFields` 中，无需单独调用评论读取工具。
> 工时取数优先级：先取主字段（分钟），缺失或为 0 时再用兜底字段换算，二者都缺则留空。

### 第五步：读取并提炼 Notion 模板结构

```
API-retrieve-page-markdown({ page_id: "3615667c6d3a80089c1aeceaebf33711" })
```

- **只提取结构骨架与填写原则**，剥离模板中所有"模板：""示例："占位正文（如企业 SaaS 平台、数据中心等示例数据）。
- 保留三大板块：
  1. **本周工作进展**（项目类按项目分组；非项目类按标签汇总）
  2. **风险与问题**
  3. **下周工作计划**
- 保留填写原则：写结果与说明（非"做了什么"流水账）、可举证事实、指标待上线可注明。

### 第六步：内容组织规则

**本周工作进展**
- 项目类任务：按项目分组，每条格式 `【模块/阶段/任务】 具体任务；说明：……；状态：已完成/进度N%`。
- 非项目类/零散小任务：按标签【日常运维】【需求支持】【质量保障】【协同支撑】【效率改进】**合并汇总**，写"总量 + 1~2 个典型案例"，避免逐条流水账。
- "说明"优先取最新评论；无评论时用任务描述提炼，并可注明"指标待上线/灰度后补充"。

**风险与问题**
- 来源：阻塞状态任务、评论中含风险/阻塞关键词的任务。
- 格式：`【模块/任务】 现象：…；影响：…；需要：…；期望解决时间：…`。
- **若无法自动识别风险，保留板块标题并写"（本周无重大风险，或待人工补充）"，不要整段省略。**

**下周工作计划**
- 来源：第三步 C 组（dueDate 落在下周）的任务。
- 格式：`【模块/任务】 具体任务；截止时间：<dueDate 北京时间>；完成标志：…`。
- "完成标志"Teambition 通常无现成字段：优先从任务描述/评论提炼；无法提炼时留"完成标志：待补充"。

### 第七步：写入前自检（对应模板"30 秒自检"）

生成内容后、写入 Notion 前逐项校验，未通过则回填提示占位：
- [ ] 每条进展是否写了"说明"而非仅"做了什么"？
- [ ] 风险项是否写清"需要谁支持 + 时间点"？
- [ ] 下周计划是否写清"截止时间 + 完成标志"？
- [ ] 是否已剥离模板示例正文？
- [ ] 小任务是否已按标签合并、无流水账？

### 第八步：幂等检查 + 创建并写入周报页面

1. **幂等检查**：先查目标父页面下是否已存在 `软件研发周报 <weekStart> · 童美娜`。
   - 已存在：提示用户选择"更新现有页面"或"另建副本"，默认不重复创建。
   - 不存在：继续创建。

2. **创建页面**：
```javascript
API-post-page({
  parent: { page_id: "3615667c6d3a80de9cc1fdb63d3424d4" },
  properties: { title: [{ text: { content: "软件研发周报 <weekStart> · 童美娜" } }] },
  children: [...]
})
```

3. **分批写入内容块**（每批 ≤100 个 block）：
```javascript
API-patch-block-children({
  block_id: "<新建页面ID>",
  children: [
    { type: "heading_2", heading_2: { rich_text: [{ text: { content: "本周工作进展" } }] } },
    { type: "bulleted_list_item", bulleted_list_item: { rich_text: [...] } }
  ]
})
```

4. **空结果跳过**：某项目本周对她无任务时跳过该项目，不生成空标题段。

## 输出

成功时返回：
- `notionPageId`
- `notionUrl`
- `weekStart` / `weekEnd`
- `projectCount`（纳入的项目数）
- `taskStats`（已完成 / 进行中 / 下周计划 各计数）
- `reportStatus = success`

失败时返回：
- `reportStatus = failed`
- `error`（原始错误摘要）
- `writtenPartial`（已写入的 block 数，供续写/重试）

## 失败策略

- Teambition 项目/任务查询失败：终止，不写 Notion。
- 某字段（如 `updated`）TQL 不可查询：降级为先拉任务再在 detail 阶段按时间过滤。
- Notion 读取模板失败：使用默认结构（本周工作进展 / 风险与问题 / 下周工作计划）继续执行。
- Notion 写入失败：返回已写入部分和失败原因，供续写重试。
- 幂等命中：默认暂停并询问，不静默重复创建。

## 工时换算

- 毫秒 → 分钟：`÷ 60000`
- 毫秒 → 小时：`÷ 3600000`
- 实际工时取数优先级：主字段（分钟）> 兜底字段（毫秒换算）> 留空

## 变更记录（相对 v2）

1. **新增时间窗口**：TQL 按"本周已完成 / 本周进行中 / 下周计划"三组分别筛选，修复历史在办任务全量混入问题。
2. **补全「下周工作计划」数据来源**：按 dueDate 落在下周筛选，映射截止时间与完成标志。
3. **补全「风险与问题」来源**：阻塞任务 + 评论关键词；无识别结果保留板块并提示人工补充。
4. **模板去示例**：明确只提取结构与填写原则，剥离示例正文。
5. **小任务按标签合并**：避免流水账，写"总量 + 典型案例"。
6. **工时双字段优先级**：主字段优先、兜底换算。
7. **项目翻页**：get_user_projects 循环翻页。
8. **幂等检查**：当周同名页面先查重。
9. **空结果跳过**：无任务项目不生成空段。
10. **写入前自检**：内置模板的 30 秒自检校验。
11. **时间范围可选**：默认本周一~本周日（Asia/Shanghai）。
12. **优先级与参与者**：优先级用于排序/紧急标记；可选纳入参与者任务。
