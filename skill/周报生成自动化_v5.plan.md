# Weekly Report — 周报生成（v5）

## 目标定位

- 用户说"生成周报""写周报""本周总结""周工作日志"等意图时触发。
- 动态拉取所有进行中项目（`isArchived=false AND isSuspended=false`），数量不写死、支持翻页。
- **执行人 ID 与项目自定义字段 ID 运行时动态解析**；Notion 页面 ID 固定（文档不变更）。
- 严格区分三个时间维度：**本周已完成 / 本周进行中 / 下周计划**。

## 配置（语义锚点）

```yaml
# —— 身份解析（支持自评 + 批量代生成）——
identity:
  mode: current_user        # current_user | byName | byExecutorIds
  byName: ~                 # 如 童美娜（mode=byName 时用）
  byExecutorIds: ~          # 如 [id1, id2, ...]（mode=byExecutorIds，团队批量，每人一份周报）
  overrideExecutorId: ~     # 仅当自动解析失败时手动兜底
  # 解析工具：见“第零步”——必须先 listTools 确认真实工具名，不可凭“类工具”猜测

fieldHints:                 # 按“名称/类型”逐项目解析，不写死字段 ID
  实际工时: { type: worktime, prefer: "worktime:total:worktime" }
  计划工时: { type: worktime, prefer: "worktime:total:plantime" }
  最新评论: { nameContains: 评论 }

riskKeywords:               # 风险/阻塞识别词，可调整（避免漏报/误报）
  - 阻塞
  - 延期
  - 卡住
  - 风险
  - 受阻
  - 等待
  - 需要支持
  - 依赖未就绪

limits:                     # 批量与性能护栏
  projectPageSize: 50       # 项目翻页，循环直到取完
  detailBatchSize: 25       # get_task_detail 每批最多 25 个 ID
  taskWarnThreshold: 500    # 总任务数超过则告警并提示收窄范围
  taskHardCap: 2000         # 超过硬上限时分段处理或要求缩小时间/项目范围

# —— 固定值（Notion 文档不会变更，直接写死）——
notion:
  templatePageId: "3615667c6d3a80089c1aeceaebf33711"   # 软件研发周报模板
  targetParentId: "3615667c6d3a80de9cc1fdb63d3424d4"   # 周报目标父页面
```

## MCP 调用前置

调用任何 MCP 工具前，先 `listTools` 读取真实工具名与 schema，确认参数结构与字段名，**严禁凭“类工具”模糊描述或记忆传参**。

## 执行流程

### 第零步：环境与身份解析（先确认工具，再解析身份）

1. **确认可用工具（关键，对应批判①）**：
   - 先 `listTools`，在结果中**按能力**定位真实工具：
     - 当前用户工具：名称/描述含 `me`/`current user`/`whoami`/`当前用户`/`账户信息` 的只读工具。
     - 项目成员工具：可按项目列成员、支持姓名匹配的工具。
     - 项目自定义字段定义工具：可列出项目 customFields 定义（含 `_id/name/type`）的工具。
   - **把解析到的真实工具名记录到运行上下文**，后续步骤引用该名，而不是写死或猜测。
   - 若 `current_user` 能力的工具在本 MCP 中**不存在**：自动降级到 `byName`；`byName` 也不可用时要求填 `overrideExecutorId`；仍无则 **fail fast** 并明确报“无法确定执行人：缺少当前用户工具，请改用 byName 或填 overrideExecutorId”。

2. **解析执行人（对应批判②，支持三种模式）**：
   - `current_user`：调上一步确认的当前用户工具拿登录者 ID（自评，执行人=调用者）。
   - `byName`：用项目成员工具按姓名匹配（`姓名 → memberId`）；命中多个时回列候选要求确认。
   - `byExecutorIds`：传入多个执行人 ID，**对每个执行人各生成一份独立周报**（团队 lead 批量代生成场景）。
   - **作用域声明**：本 plan 处理“自评 + 按执行人批量代生成”；不含“组评/上级代评打分”等评价类逻辑（如需，后续版本扩展）。

3. **时间窗口**：未指定则按 Asia/Shanghai 计算本周一~本周日及下周一~下周日，输出 `weekStart/weekEnd/nextWeekStart/nextWeekEnd`。

> 多执行人时，第一~第八步对每个 executorId 各跑一遍，分别产出页面。

### 第一步：拉取进行中项目（翻页，对应批判④之一）

```javascript
get_user_projects({ pageSize: limits.projectPageSize, nextToken: <上一页token> })
// 循环翻页直到无 nextToken；过滤 isArchived=false 且 isSuspended=false
```

### 第二步：逐项目解析自定义字段映射（动态解析 customFields）

> Teambition 自定义字段 ID 按项目定义，不同项目不同，不能用一个 ID 套所有项目。

对每个进行中项目，用第零步确认的“字段定义工具”读取字段定义，按 `fieldHints` 的名称/类型匹配出真实字段 ID，构建并缓存 `fieldMap[projectId]`：

```
fieldMap[projectId] = {
  实际工时: <type=worktime 且名含“实际工时”；优先系统键 worktime:total:worktime>,
  计划工时: <同上，优先 worktime:total:plantime>,
  最新评论: <名称含“评论”的文本字段 ID>
}
```

- 工时优先用跨项目稳定的系统键，项目级字段仅作兜底。
- 某项目某字段**解析不到 → 该字段在该项目留空并记入告警**，**绝不跨项目套用别的 ID**。

### 第三步：按时间窗口分类查任务

对每个进行中项目执行三组带时间条件的 TQL（`<executorId>` 为第零步解析结果）：

```
# A. 本周已完成
projectId = <项目ID> AND executorId = <executorId> AND isArchived = false
  AND isDone = true AND accomplishTime >= <weekStart> AND accomplishTime <= <weekEnd>
# B. 本周进行中
projectId = <项目ID> AND executorId = <executorId> AND isArchived = false
  AND isDone = false AND progress > 0 AND updated >= <weekStart> AND updated <= <weekEnd>
# C. 下周计划
projectId = <项目ID> AND executorId = <executorId> AND isArchived = false
  AND isDone = false AND dueDate >= <nextWeekStart> AND dueDate <= <nextWeekEnd>
```

> 字段可查询性以 schema 为准；不可查询时降级到 detail 阶段按时间过滤。

### 第四步：批量读取详情（含数量护栏，对应批判④）

- **去重汇总任务 ID**：合并 A/B/C 三组并去重，得到总任务数 `N`。
- **数量护栏**：
  - `N > limits.taskWarnThreshold(500)`：输出性能告警，提示用户可缩小时间范围或项目集；继续执行。
  - `N > limits.taskHardCap(2000)`：分段处理（按项目或时间切片）或要求缩小范围，避免超时。
- **分批串行拉取**：`get_task_detail` **每批 ≤25 个 ID，按批串行**（避免并发打爆 MCP/限流）；批间可短暂退避，失败批次记录并重试一次。
- 读取字段（取值通过 `fieldMap` 动态定位，不引用硬编码 ID）：

| 字段 | 用途 |
|------|------|
| `content` | 任务名称 |
| `priority` | 优先级（2=非常紧急,1=紧急,0=普通,-10=较低）→ 排序/紧急标记 |
| `progress` | 进展百分比 |
| `isDone` | 完成状态 |
| `dueDate` | 截止日期（UTC +8h） |
| `accomplishTime` | 完成时间（UTC +8h） |
| `tags` | 小任务按标签合并 |
| `customFields[ fieldMap[pid].最新评论 ]` | 最新评论（说明/风险来源） |
| `worktime:total:worktime`（首选）/ 兜底字段 | 实际工时 |
| `worktime:total:plantime`（首选）/ 兜底字段 | 计划工时 |

### 第五步：读取并提炼 Notion 模板结构

`API-retrieve-page-markdown({ page_id: notion.templatePageId })`，只提取三大板块骨架（本周工作进展 / 风险与问题 / 下周工作计划）与填写原则，**剥离“模板：/示例：”占位正文**。

### 第六步：内容组织规则

**本周工作进展**
- 项目类：按项目分组，`【模块/阶段/任务】 具体任务；说明：…；状态：已完成/进度N%`。
- 非项目/零散小任务：按标签【日常运维】【需求支持】【质量保障】【协同支撑】【效率改进】**合并汇总**，写“总量 + 1~2 个典型案例”。
- “说明”优先取最新评论；无评论时用任务描述提炼。

**风险与问题（风险词已定义，对应批判③）**
- 来源：阻塞状态任务，或评论/描述中命中 `riskKeywords` 的任务。
- 命中判定：文本包含任一关键词即纳入（可在 `riskKeywords` 增删以调松紧，降低漏报/误报）。
- 格式：`【模块/任务】 现象：…；影响：…；需要：…；期望解决时间：…`。
- 无命中：保留板块并写“（本周无重大风险，或待人工补充）”，不整段省略。

**下周工作计划**
- 来源：第三步 C 组（dueDate 落在下周）。
- 格式：`【模块/任务】 具体任务；截止时间：<dueDate 北京时间>；完成标志：…`。
- “完成标志”优先从描述/评论提炼，无法提炼时留“待补充”。

### 第七步：写入前自检

- [ ] 每条进展是否写了“说明”而非仅“做了什么”？
- [ ] 风险项是否写清“需要谁支持 + 时间点”？
- [ ] 下周计划是否写清“截止时间 + 完成标志”？
- [ ] 是否已剥离模板示例正文？
- [ ] 小任务是否已按标签合并、无流水账？
- [ ] 是否有项目字段解析失败而留空？（有则列出告警）
- [ ] 任务总数是否触发告警阈值？（触发则在输出中提示）

### 第八步：幂等检查 + 写入固定父页

1. **幂等**：在 `notion.targetParentId` 下查是否已存在 `软件研发周报 <weekStart> · <执行人姓名>`；存在则提示“更新/另建副本”，默认不重复创建。
2. **创建页面**：
```javascript
API-post-page({
  parent: { page_id: notion.targetParentId },
  properties: { title: [{ text: { content: "软件研发周报 <weekStart> · <执行人姓名>" } }] },
  children: [...]
})
```
3. **分批写入**（每批 ≤100 个 block）。
4. **空结果跳过**：某项目本周对该执行人无任务则跳过，不生成空标题段。

## 输出

成功（每个执行人各一条）：`notionPageId / notionUrl / executorId / executorName / weekStart / weekEnd / projectCount / taskStats（已完成/进行中/下周计划）/ taskTotal / fieldResolveWarnings / volumeWarning（是否触发数量告警）/ reportStatus=success`。
失败：`reportStatus=failed / error / writtenPartial`。

## 失败策略

- 缺少当前用户工具且无 byName/override：fail fast，明确报因。
- Teambition 项目/任务查询失败：终止，不写 Notion。
- 某项目某自定义字段解析失败：留空 + 告警，不跨项目套 ID，不中断整体。
- 任务数超硬上限：分段或要求缩小范围，避免超时。
- Notion 读取模板失败：用默认结构继续。
- Notion 写入失败：返回已写入部分与原因，供续写重试。

## 工时换算

- 毫秒 → 分钟 `÷60000`；毫秒 → 小时 `÷3600000`。
- 实际工时取数优先级：系统键 worktime:total:worktime > 项目字段兜底 > 留空。

## 变更记录（相对 v4，对应图片 4 条批判）

1. **工具存在性确认（批判①）**：第零步先 `listTools` 按能力定位真实工具名并记录，不再用“当前用户/whoami 类工具”这种模糊表述；工具不存在时定义了降级与 fail fast 路径。
2. **批量代生成补全（批判②）**：identity 新增 `byExecutorIds` 模式，支持团队 lead 为多人各生成一份周报；并明确作用域为“自评 + 代生成”，不含评价/打分逻辑。
3. **风险关键词定义（批判③）**：新增 `riskKeywords` 配置块并给出默认词表与命中判定，消除“风险来源未定义”的漏报/误报。
4. **批量数量护栏（批判④）**：新增 `limits` 配置；第一步项目翻页、第四步去重统计总数、分批串行拉取、告警阈值(500)与硬上限(2000)处理，明确并发与截断策略。
