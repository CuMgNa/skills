# Notion 富格式测试报告编排
> 本技能输出**增强 Markdown**（含 Notion 特有标记），由下游 `notion-test-report` 写入 Notion 页。

> **v2 标准管线（已落地）**：实际 Notion 写入由 `mcp/scripts/publish_report.py` + `lib/report_templates/standard.py` 完成，渲染**八段标准模板**：①测试结论 ②指标看板 ③重点问题（动态聚类）④功能测试范围与执行情况 ⑤模块明细 ⑥风险评估 ⑦未解决与待回归清单 ⑧附录（解析详情/冲突/阈值，toggle 折叠）。本文档下述各节规则作为模板内容口径来源；解析层为 `lib/material_context.py`（旧 `lib/test_plan_material.py` 已转兼容 shim），缺陷语义来自 `lib/bug_semantic_context.py`（读 `mcp/output/bug-semantic/*.jsonl`）。集中配置见 `lib/report_config.py`，多语言见 `lib/report_templates/strings.py`（`--locale zh-CN|en-US`）。展示级别只认 bugStats，severity 仅作语义参考、不反向污染数字。

## 第一节「测试结果」生成规则

**数字全部取自 `bugStats`，禁止重新计数。格式与 `test-report` 技能完全对齐（钉钉版同一节）**。

> ⚠️ **分组维度硬约束（必须遵守）**：第一节的 bullet **必须按【业务问题方向 / 主题】归纳**（可跨模块合并同类缺陷，例如把多条消息通知缺陷归纳为「消息通知触达 / 扣费逻辑」、把群聊相关缺陷归纳为「求救群聊与会话状态同步」）。**严禁按级别（一级 / 二级 / 三级 / 四级）分组，也严禁按状态（激活-待确认 / 激活-已确认 / 已延期）分组**——按级别 / 状态分组是第三节的职责。级别信息只允许在量化摘要句中以「（一级 X、二级 X、三级 X、四级 X）」形式出现一次。

### 步骤1：归纳问题分类

基于 `bugStats.未关闭列表`（状态为激活-待确认或激活-已确认），按标题前缀提取模块分布，归纳问题类型：

- 提取各缺陷的【模块】前缀，合并同类（如【消息通知】×8、【停港逻辑】×2）
- 二级及以上缺陷必须单独点名，格式：`**{个数} 个高优问题（二级及以上）需优先修复**`
- 已延期缺陷说明：`**{个数} 个已延期问题建议下个版本跟进**`
- 无未解决缺陷时写「本轮无遗留未解决缺陷」，并简述已解决情况

### 步骤2：生成测试结果（与 test-report 格式完全一致）

```markdown
本轮测试已完成 {bugStats.byModule-keys.length} 个模块的功能验证，当前仍有 **{bugStats.byStatus.未解决} 个未解决缺陷**，主要集中在 **{【分类1】/【分类2】/…}** 等方向，其中 **{二级及以上缺陷数} 个高优问题（二级及以上）需优先修复**，**{已延期数} 个已延期问题建议下个版本跟进。**

- **【分类1】**：根据未解决缺陷归纳的具体描述
- **【分类2】**：根据未解决缺陷归纳的具体描述
```

### 渲染判定（分两档，不再整段隐藏）

> **有可用辅助测试资料且能解析出测试范围表** → 渲染完整执行表（含核心测试点 / 优先级）。<br>**拿不到辅助资料（未上传 / 读取失败 / 表头不对 / 全空）** → 降级渲染「精简执行表」：仅用 `bugStats.byModule` 如实呈现 模块 / 未解决 / 待回归 / 结果状态，不含核心测试点与优先级列。<br>精简表只如实呈现 byModule，**不推断、不占位、不报错**。<br>**降级时必须在第二节顶部补一个黄色 callout** 显式标注降级原因（资料类型 + 解析原因），避免用户不知道为什么没看到完整表。

### 模块别名映射（解决资料模块 ≠ bugStats.byModule key 的问题）

禅道缺陷标题前缀（如「对讲群-群信息」）经常比资料里的模块名更细分，脚本按以下三层匹配归并：

1. **精确相等**：`zentao_mod == plan_module`
2. **别名归并**：通过 `MODULE_ALIAS` 表把禅道前缀映射到规范模块名（兼容磐钴/SaaS1期 + 天翼/星联2期）
3. **前缀细分匹配**：`对讲群-群昵称` 归到 `对讲群-群信息` 行（取最长前缀命中，避免重复计数）

**禅道 key → 资料模块行 的归并必须唯一**（一个禅道 key 只能贡献给一个资料行），否则会出现数字虚高。

### 执行表生成算法

```plain text
输入：bugStats.byModule（已按别名映射归一化），辅助测试资料文本（可选）

【第一阶段：辅助资料解析（lib/material_context.py，多资料合并）】
1. 资料类型识别（detect_material_kind）：test_plan / logic_outline / prd / unknown
2. 多级锚点扫描：
   - 强锚点：1.4.1 / 2.1 范围 / 范围全景表 / 测试范围表 / 全景表
   - 弱锚点：测试方案 / 测试计划 / 测试大纲 / 逻辑大纲 / 功能清单 / 模块清单 / 用例矩阵 / 测试矩阵
3. 全文表扫描：HTML <table> + Markdown |...| 表，定位含 module 列的表头
4. 表头同义词匹配：核心测试点 ↔ 测试点 ↔ 测试方向 ↔ 测试范围
5. 输出 (rows, title, material_kind, parse_reason) 四元组

【第二阶段：完整执行表（含核心测试点/优先级）】
1. 以解析得到的资料行为行模板（模块 + 核心测试点 + 优先级）
2. **禅道 key → 资料模块行 的归并必须唯一**：
   - 先建立 zentao_to_plan 映射（每个禅道 key 只匹配它能命中的最长 plan_module）
   - 同一禅道 key 不会被两个资料行重复计数
3. 每行从 byModule[模块] 取数：未关闭 / 待回归 / 回归不通过 / 延期
4. 结果状态按确定性判定链推导（见各列填充规则）
5. byModule 有、资料无对应行 → 追加行，备注「资料外模块」
6. 资料有、byModule 无缺陷 → 已覆盖填「✅ 通过」，未覆盖不出现
7. 解析状态写入 meta.section2_parse_reason，供降级回放

【第三阶段：精简执行表（降级，不再整段隐藏）】
1. 触发条件：parse_reason ∈ {no_text, no_marker, no_table, no_header, no_header_or_empty}
2. 直接遍历 bugStats.byModule.keys() 作为行
3. 列：测试模块 / 结果状态 / 未解决 / 待回归 / 备注
4. 仅如实呈现 byModule，不补核心测试点与优先级
5. 第二节顶部补黄色 callout：
   - 资料类型（test_plan / logic_outline / prd / unknown）
   - 解析结果（render_parse_notice(reason)）
   - 处置说明（已降级为精简表）
   - 建议（提供含「测试模块 / 核心测试点 / 优先级」的标准表格可恢复完整表）
6. 不推断、不占位、不报错
```

## 第三节「未解决问题汇总」生成规则

**数据全取 `bugStats.未关闭列表`，不许改写标题，全部罗列、不截断（与钉钉版口径一致）。**

```markdown

**激活-已确认（回归不通过）（{回归不通过数}）**
1. [{级别}] {标题（原文)}
2. [{级别}] {标题（原文)}
……（完整列表）

**激活-待确认（{激活待确认数}）**
1. [{级别}] {标题（原文)}
……

**已延期（{已延期数}）**
1. [{级别}] {标题（原文)}

```

排序规则：同组内按优先级从高到低（一级→二级→三级→四级）。

## 第四节「待回归」生成规则

**数据全取 `bugStats.待回归列表`（状态=已解决）。**

```markdown
## 四、待回归

{待回归列表.length} 个已解决缺陷待 QA 回归验证：

1. [{级别}] {标题（原文)}
2. [{级别}] {标题（原文)}
……
```

## 第五节「风险与遗留影响评估」生成规则

**评估对象**：仅 `bugStats` 中状态为「未解决（激活）」与「已延期」的缺陷；已解决 / 待回归不评。**按业务方向聚合**（复用第一节的业务方向归纳）：一级 / 二级缺陷必须单独成行，三级 / 四级按方向归并。

### 影响面归类（从缺陷标题 / 模块前缀提取，不自创）

分为 **计费 / 数据一致性 / 安全 / 触达 / 体验** 五类：命中关键词归入对应类，命中多类时取最高风险类。

### 建议处置档位（与风险等级一一对应）

- 🔴 高 → 阻断上线，优先修复后进入下一轮回归
- 🟠 中 → 下轮回归前修复，不阻断但需跟踪
- 🟡 低 → 可下个版本跟进 / 视情况接受

### 约束

- 缺陷数量、级别、标题、模块全部取自 `bugStats`，风险方向沿用第一节归纳，**不得新增 bugStats 之外的风险项**。
- 影响面、风险等级、建议处置只能由上述规则推导，**禁止臆测具体业务后果或编造数字**。
- 无未解决 / 延期缺陷时如实写「本轮无高风险遗留」，不强造。

## 关键约束

1. **数字全取 bugStats**：第一节、第三节、第四节、第五节的每一个数字都必须从 `bugStats` 取，禁止重算
2. **标题不许改写**：`未关闭列表` 和 `待回归列表` 中的标题取 `bugStats` 原文
3. **第二部分分两档渲染**：有资料出完整执行表；无资料降级出精简执行表（仅如实呈现 bugStats.byModule，不推断、不占位、不报错），不再整段隐藏
4. **模块别名映射必须应用**：`bugStats.byModule` 的 key 是归一化后的模块名，用于拼表
5. **降级必须显式标注**：若任何环节失败导致版式不完整，必须在报告顶部先加普通 Markdown 可见块（`## ⚠️ 版式降级通知` + 加粗说明），可选再由写入层追加黄色 callout；不得只用 callout 承载降级提示
6. **关键 callout 不只依赖 Markdown**：报告信息、功能测试结论、版式降级通知等关键卡片必须带 `NOTION_CALLOUT:*` 语义占位，交由 `notion-test-report` 优先转成强类型 callout block
7. **Notion Markdown 语法自检**：列表项 `-` 后必须有空格；callout / toggle / 列表子内容用 Tab 缩进；不得把最终报告正文包进代码围栏；不得残留未被解析的 callout 原始标签
8. **严禁幻觉 / 编造（最高优先级）**：报告全部内容（缺陷数量、级别、状态、标题、模块、结论）**必须严格基于已有资料**（`bugStats`、禅道汇总、用户上传的测试方案 / 大纲）。**不得自行编造、推断或补充资料中不存在的缺陷、数字或结论**；资料里没有的就不写。无未解决缺陷时如实写「本轮无遗留未解决缺陷」，不得臆造。
9. **第一节只按业务方向归纳**：严禁按级别或状态分组（见「第一节『测试结果』生成规则」的分组维度硬约束）。
10. **辅助资料解析必须健壮**：脚本不再写死 1.4.1 全景表。多级锚点扫描（强锚点 + 弱锚点）+ 同义表头识别 + 资料类型自识别；解析失败时必须把失败原因（reason）和资料类型（kind）显式标注到降级 callout，不得静默返回空 rows。
11. **禅道 key 归并必须唯一**：同一禅道 key 只能贡献给一个资料模块行，否则会出现数字虚高。`build_execution_rows` 通过"先建立 zentao_to_plan 映射、取最长 plan_module 命中"保证唯一性。
12. **重点问题动态聚类**：第②段「重点问题」由 `lib/key_issues.py` 按缺陷 `impactSignals`（资金/计费、权限/安全、数据一致性、消息/通信、界面/体验）动态成组，而非写死分类；方向成立需满足支撑数阈值或含二级及以上。每条重点问题须带缺陷 ID 可溯源（C3'），不可溯源的业务影响标注「（影响待复核）」，不得作确定性结论输出。
13. **缺陷语义只读消费、级别不被 severity 污染**：`BugSemanticContext` 三来源（持久化产物 → 禅道 steps → 标题）按固定优先级 reconcile，冲突写入附录；展示级别恒等于 `bugStats.byLevel`（C4'）。报告阶段**绝不触发**缺陷创建。
