# Notion 富格式测试报告编排

> 本技能输出**增强 Markdown**（含 Notion 特有标记），由下游 `notion-test-report` 写入 Notion 页。

## 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| `bugStats.json` | `bug-stats` 技能产出 | 单一统计事实源，**数字全部从此处取** |
| 辅助测试资料（可选） | 用户上传的 Notion 页链接 | 测试方案 / 测试大纲 |
| handoff 配置 | `qa-agent-report-publish` 传入 | 报告元信息 |

## 固定配置

```yaml
notion:
  templatePageId: "c1b23699-3b3b-4b06-b2ac-0ec9ede194b6"  # 样板页
  materialRootPageId: "3585667c-6d3a-8084-bb07-d7a30a3945d9"  # 测试资料
  outlineParentPageId: "3665667c-6d3a-8079-85ff-ec1a76e55946"  # 测试大纲
```

## 输出格式（严格四节）

```markdown
<callout icon="📝" color="gray_bg">
  **报告信息**
  - 项目：{handoff.项目名}
  - 测试类型：{handoff.reportMeta.testType}
  - 覆盖期：{handoff.reportMeta.coverageStart}～{handoff.reportMeta.coverageEnd}
  - 测试人：{handoff.reportMeta.tester}
  - 开发者：{handoff.reportMeta.developer}
  - 覆盖模块：{bugStats.byModule-keys.join(', ')}
</callout>

## 一、测试结果
{量化摘要 + 分类bullet，数字全部取自bugStats}

<!-- 二、功能测试范围与执行情况：条件模块，见§第二部分 -->

## 三、未解决问题汇总
{按状态分组 + 编号列表，不截断，数据全取bugStats.byStatus}

## 四、待回归
{待回归列表，状态=已解决}
```

## 第一节「测试结果」生成规则

**数字全部取自 `bugStats`，禁止重新计数。格式与 `test-report` 技能完全对齐（钉钉版同一节）**。

### 步骤1：归纳问题分类

基于 `bugStats.未解决列表`（状态含激活/延期），按标题前缀提取模块分布，归纳问题类型：

- 提取各缺陷的【模块】前缀，合并同类（如【消息通知】×8、【停港逻辑】×2）
- 二级及以上缺陷必须单独点名，格式：`**{个数} 个高优问题（二级及以上）需优先修复**`
- 已延期缺陷说明：`**{个数} 个已延期问题建议下个版本跟进**`
- 无未解决缺陷时写"本轮无遗留未解决缺陷"，并简述已解决情况

### 步骤2：生成测试结果（与 test-report 格式完全一致）

```markdown
本轮测试已完成 {bugStats.byModule-keys.length} 个模块的功能验证，当前仍有 **{bugStats.byStatus.未解决} 个未解决缺陷**，主要集中在 **{【分类1】/【分类2】/…}** 等方向，其中 **{二级及以上缺陷数} 个高优问题（二级及以上）需优先修复**，**{已延期数} 个已延期问题建议下个版本跟进。**

- **【分类1】**：根据未解决缺陷归纳的具体描述
- **【分类2】**：根据未解决缺陷归纳的具体描述
```

## 第二部分「功能测试范围与执行情况」——纯条件模块（关键规则）

### 条件渲染判定

> **只要拿不到可用的辅助测试资料（用户未上传 或 上传后读取失败），就跳过并隐藏整个第二部分**。
> 不推断、不占位、不报错。后续小节（三、四）顺延。

### 当第二部分存在时

1. **灰色 quote**：
   ```markdown
   > 本轮依据《{测试方案标题}》中「{模块}」范围及缺陷回归计划执行……
   ```

2. **5列执行表**（`header-row="true"`，列宽 `43 / 150 / 400 / 63 / 175`）：

   | # | 测试模块 | 核心测试点 | 优先级 | 执行结果 |
   |---|----------|-----------|--------|----------|
   | 1 | 🔐 登录与认证 | 账号鉴权、Token、心跳保活、多角色登录权限 | **P0** | ✅ 已修复待回归（1） |
   | 2 | 📦 设备与分组管理 | 设备绑定、分组层级、设备标签、归属逻辑 | **P1** | ⚠️ 2 个未解决（电子围栏、订阅通知） |

3. **执行表「执行结果」列填充规则**（全部取自 `bugStats.byModule`）：

   | 情形 | 填充 |
   |------|------|
   | 该模块仅有已解决缺陷 | `✅ 已修复待回归（N）` |
   | 该模块有激活/延期缺陷 | `⚠️ X 个未解决（模块名列表）` |
   | 该模块本轮覆盖且无未关闭缺陷 | `✅ 通过` |
   | 无法归类的缺陷 | `⚠️ X 个未解决（未分类）` |
   | 括号附注回归不通过数、已延期数 | 如 `（含回归不通过 1）` |

4. **green callout 结论**：
   ```markdown
   <callout icon="✅" color="green_bg">
     **功能测试结论**：本轮覆盖 {N} 个模块，其中 {M} 个已修复待回归，{K} 个存在未关闭缺陷需继续跟进……
   </callout>
   ```

### 执行表生成算法

```
输入：bugStats.byModule（按模块别名映射归一化后的统计）
1. 以辅助资料的测试方案 1.4.1 全景表为行模板（模块清单 + 核心测试点 + 优先级）
2. 仅保留本轮实际涉及模块：bugStats.byModule.keys() ∪ 资料当前阶段任务
3. 对每个模块查 bugStats.byModule[模块名] 填充执行结果
4. 若 bugStats.byModule 中有模块但资料中无对应行 → 追加一行（标注「资料外模块」）
5. 若资料中有模块但 bugStats.byModule 中无缺陷 → 查测试计划确认是否本轮覆盖：
   - 已覆盖 → 执行结果填「✅ 通过」
   - 未覆盖 → 不出现在执行表中
```

## 第三节「未解决问题汇总」生成规则

**数据全取 `bugStats.未解决列表`，不许改写标题，不截断。**

```markdown
## 三、未解决问题汇总

**激活-已确认（回归不通过）（{回归不通过数}）**
1. [{级别}] {标题（原文）}
2. [{级别}] {标题（原文）}
……（完整列表）

**激活-待确认（{激活待确认数}）**
1. [{级别}] {标题（原文）}
……

**已延期（{已延期数}）**
1. [{级别}] {标题（原文）}
……
```

排序规则：同组内按优先级从高到低（一级→二级→三级→四级）。

## 第四节「待回归」生成规则

**数据全取 `bugStats.待回归列表`（状态=已解决）。**

```markdown
## 四、待回归

{待回归列表.length} 个已解决缺陷待 QA 回归验证：

1. [{级别}] {标题（原文）}
2. [{级别}] {标题（原文）}
……
```

## 关键约束

1. **数字全取 bugStats**：第一节、第三节、第四节的每一个数字都必须从 `bugStats` 取，禁止重算
2. **标题不许改写**：`未解决列表` 和 `待回归列表` 中的标题取 `bugStats` 原文
3. **第二部分条件渲染**：无资料必须隐藏，不能推断造表
4. **模块别名映射必须应用**：`bugStats.byModule` 的 key 是归一化后的模块名，用于拼表
5. **降级必须显式标注**：若任何环节失败导致版式不完整，必须在报告顶部加 `<callout icon="⚠️">降级说明：……</callout>`
