> 本技能输出**增强 Markdown**（含 Notion 特有标记），由下游 `notion-test-report` 写入 Notion 页。

## 第一节「测试结果」生成规则

**数字全部取自 `bugStats`，禁止重新计数。格式与 `test-report` 技能完全对齐（钉钉版同一节）**。

> ⚠️ **分组维度硬约束（必须遵守）**：第一节的 bullet **必须按【业务问题方向 / 主题】归纳**（可跨模块合并同类缺陷，例如把多条消息通知缺陷归纳为「消息通知触达 / 扣费逻辑」、把群聊相关缺陷归纳为「求救群聊与会话状态同步」）。**严禁按级别（一级 / 二级 / 三级 / 四级）分组，也严禁按状态（激活-待确认 / 激活-已确认 / 已延期）分组**——按级别 / 状态分组是第三节的职责。级别信息只允许在量化摘要句中以「（一级 X、二级 X、三级 X、四级 X）」形式出现一次。

### 步骤1：归纳问题分类

基于 `bugStats.未关闭列表`（状态为激活-待确认或激活-已确认），按标题前缀提取模块分布，归纳问题类型：

- 提取各缺陷的【模块】前缀，合并同类（如【消息通知】×8、【停港逻辑】×2）
- 二级及以上缺陷必须单独点名，格式：`**{个数} 个高优问题（二级及以上）需优先修复**`
- 已延期缺陷说明：`**{个数} 个已延期问题建议下个版本跟进**`
- 无未解决缺陷时写「本轮无遗留未解决缺陷」，并简述已解决情况

### 步骤2：生成测试结果（与 test-report 格式完全一致）

```markdown
本轮测试已完成 {bugStats.byModule-keys.length} 个模块的功能验证，当前仍有 **{bugStats.byStatus.未解决} 个未解决缺陷**，主要集中在 **{【分类1】/【分类2】/…}** 等方向，其中 **{二级及以上缺陷数} 个高优问题（二级及以上）需优先修复**，**{已延期数} 个已延期问题建议下个版本跟进。**

- **【分类1】**：根据未解决缺陷归纳的具体描述
- **【分类2】**：根据未解决缺陷归纳的具体描述
```

## 第二部分「功能测试范围与执行情况」——纯条件模块（关键规则）

### 渲染判定（分两档，不再整段隐藏）

> **有可用辅助测试资料** → 渲染完整执行表（含核心测试点 / 优先级）。<br>**拿不到辅助资料（未上传 或 读取失败）** → 降级渲染「精简执行表」：仅用 `bugStats.byModule` 如实呈现 模块 / 未解决 / 待回归 / 结果状态，不含核心测试点与优先级列。<br>精简表只如实呈现 byModule，**不推断、不占位、不报错**。

### 当第二部分存在时

1. **灰色 quote**：
	```markdown
> 本轮依据《{测试方案标题}》中「{模块}」范围及缺陷回归计划执行……
	```

2. **结构化执行表**（`header-row="true"`）：
	<table header-row="true">
	<tr><td>#</td><td>测试模块</td><td>核心测试点</td><td>优先级</td><td>结果状态</td><td>未解决</td><td>待回归</td><td>备注</td></tr>
	<tr><td>1</td><td>🔐 登录与认证</td><td>账号鉴权、Token、心跳保活、多角色登录权限</td><td>**P0**</td><td>🟡 待回归</td><td>0</td><td>1</td><td>—</td></tr>
	<tr><td>2</td><td>📦 设备与分组管理</td><td>设备绑定、分组层级、设备标签、归属逻辑</td><td>**P1**</td><td>⚠️ 有未解决</td><td>2</td><td>0</td><td>含回归不通过 1</td></tr>
	</table>

精简表（无资料降级）省略「核心测试点 / 优先级」两列，仅保留 #/测试模块/结果状态/未解决/待回归/备注。

3. **各列填充规则**（数值列取自 `bugStats.byModule`，结果状态由数值确定性推导）：
	<table header-row="true">
	<tr><td>列</td><td>填充规则</td></tr>
	<tr><td>未解决</td><td>`byModule[模块].未解决`（含回归不通过），无则填 0</td></tr>
	<tr><td>待回归</td><td>`byModule[模块].已修复`，无则填 0</td></tr>
	<tr><td>结果状态</td><td>确定性判定链：未解决>0 或 回归不通过>0 → `⚠️ 有未解决`；否则 延期>0 → `⚠️ 延期`；否则 待回归>0 → `🟡 待回归`；否则本轮覆盖且全为 0 → `✅ 通过`</td></tr>
	<tr><td>备注</td><td>附注 `含回归不通过 N` / `延期 N`；无法归类缺陷单独成「未归类」行并触发告警，不混入其它行</td></tr>
	</table>

4. **green callout 结论**：

### 执行表生成算法

```plain text
输入：bugStats.byModule（已按别名映射归一化），辅助测试资料（可选）

【有辅助资料 → 完整执行表（含核心测试点/优先级）】
1. 以测试方案 1.4.1 全景表为行模板（模块 + 核心测试点 + 优先级）
2. 行集合 = 本轮覆盖模块（资料当前阶段任务）∪ bugStats.byModule.keys()
3. 每行从 byModule[模块] 取数：未解决 / 待回归 / 回归不通过 / 延期
4. 结果状态按确定性判定链推导（见各列填充规则）
5. byModule 有、资料无对应行 → 追加行，备注「资料外模块」
6. 资料有、byModule 无缺陷 → 已覆盖填「✅ 通过」，未覆盖不出现
7. 无法归类缺陷 → 单独「未归类」行 + 告警，不静默丢弃

【无辅助资料 → 精简执行表（降级，不再整段隐藏）】
1. 直接遍历 bugStats.byModule.keys() 作为行
2. 列：# / 测试模块 / 结果状态 / 未解决 / 待回归 / 备注
3. 仅如实呈现 byModule，不补核心测试点与优先级
4. 不推断、不占位、不报错；表顶标注「精简执行表（无测试方案资料）」
```

## 第三节「未解决问题汇总」生成规则

**数据全取 `bugStats.未关闭列表`，不许改写标题，全部罗列、不截断（与钉钉版口径一致）。**

```markdown
## 三、未解决问题汇总

**激活-已确认（回归不通过）（{回归不通过数}）**
1. [{级别}] {标题（原文)}
2. [{级别}] {标题（原文)}
……（完整列表）

**激活-待确认（{激活待确认数}）**
1. [{级别}] {标题（原文)}
……

**已延期（{已延期数}）**
1. [{级别}] {标题（原文)}
……
```

排序规则：同组内按优先级从高到低（一级→二级→三级→四级）。

## 第四节「待回归」生成规则

**数据全取 `bugStats.待回归列表`（状态=已解决）。**

```markdown
## 四、待回归

{待回归列表.length} 个已解决缺陷待 QA 回归验证：

1. [{级别}] {标题（原文)}
2. [{级别}] {标题（原文)}
……
```

## 第五节「风险与遗留影响评估」生成规则

**评估对象**：仅 `bugStats` 中状态为「未解决（激活）」与「已延期」的缺陷；已解决 / 待回归不评。**按业务方向聚合**（复用第一节的业务方向归纳）：一级 / 二级缺陷必须单独成行，三级 / 四级按方向归并。

### 输出结构

<table header-row="true">
<tr><td>风险方向</td><td>关联缺陷数</td><td>影响面</td><td>风险等级</td><td>建议处置</td></tr>
<tr><td>消息通知触达 / 扣费</td><td>8（三级）</td><td>计费准确性、通知可达</td><td>🔴 高</td><td>阻断上线，优先修复</td></tr>
<tr><td>电子围栏多级同步</td><td>3（含 1 回归不通过、1 延期）</td><td>数据一致性</td><td>🟠 中</td><td>下轮回归前修复</td></tr>
</table>

### 影响面归类（从缺陷标题 / 模块前缀提取，不自创）

分为 **计费 / 数据一致性 / 安全 / 触达 / 体验** 五类：命中关键词归入对应类，命中多类时取最高风险类。

### 风险等级判定矩阵（确定性，禁止凭感觉判级）

<table header-row="true">
<tr><td>条件</td><td>风险等级</td></tr>
<tr><td>存在一级未解决</td><td>🔴 高</td></tr>
<tr><td>存在二级未解决 或 任意回归不通过</td><td>🔴 高</td></tr>
<tr><td>三级 且 影响面属计费 / 数据一致性 / 安全</td><td>🟠 中</td></tr>
<tr><td>三级（一般） 或 四级</td><td>🟡 低</td></tr>
</table>

### 建议处置档位（与风险等级一一对应）

- 🔴 高 → 阻断上线，优先修复后进入下一轮回归
- 🟠 中 → 下轮回归前修复，不阻断但需跟踪
- 🟡 低 → 可下个版本跟进 / 视情况接受

### 约束

- 缺陷数量、级别、标题、模块全部取自 `bugStats`，风险方向沿用第一节归纳，**不得新增 bugStats 之外的风险项**。
- 影响面、风险等级、建议处置只能由上述规则推导，**禁止臆测具体业务后果或编造数字**。
- 无未解决 / 延期缺陷时如实写「本轮无高风险遗留」，不强造。

## 关键约束

1. **数字全取 bugStats**：第一节、第三节、第四节、第五节的每一个数字都必须从 `bugStats` 取，禁止重算
2. **标题不许改写**：`未关闭列表` 和 `待回归列表` 中的标题取 `bugStats` 原文
3. **第二部分分两档渲染**：有资料出完整执行表；无资料降级出精简执行表（仅如实呈现 bugStats.byModule，不推断、不占位、不报错），不再整段隐藏
4. **模块别名映射必须应用**：`bugStats.byModule` 的 key 是归一化后的模块名，用于拼表
5. **降级必须显式标注**：若任何环节失败导致版式不完整，必须在报告顶部先加普通 Markdown 可见块（`## ⚠️ 版式降级通知` + 加粗说明），可选再由写入层追加黄色 callout；不得只用 callout 承载降级提示
6. **关键 callout 不只依赖 Markdown**：报告信息、功能测试结论、版式降级通知等关键卡片必须带 `NOTION_CALLOUT:*` 语义占位，交由 `notion-test-report` 优先转成强类型 callout block
7. **Notion Markdown 语法自检**：列表项 `-` 后必须有空格；callout / toggle / 列表子内容用 Tab 缩进；不得把最终报告正文包进代码围栏；不得残留未被解析的 callout 原始标签
8. **严禁幻觉 / 编造（最高优先级）**：报告全部内容（缺陷数量、级别、状态、标题、模块、结论）**必须严格基于已有资料**（`bugStats`、禅道汇总、用户上传的测试方案 / 大纲）。**不得自行编造、推断或补充资料中不存在的缺陷、数字或结论**；资料里没有的就不写。无未解决缺陷时如实写「本轮无遗留未解决缺陷」，不得臆造。
9. **第一节只按业务方向归纳**：严禁按级别或状态分组（见「第一节『测试结果』生成规则」的分组维度硬约束）。
