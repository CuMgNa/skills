# Bug 统计事实源

> 本技能为 qa-agent-report-publish 的共享统计步骤,产出结构化事实对象 `bugStats.json`,钉钉报告与 Notion 报告**只消费、不重算**。

## 输入
来自上游 `zentao-bug-summary` 的输出(MD 或 JSON):
- 项目名 + 日期的 JSON 文件(`{项目}-bugs-{日期}.json`)
- 或已内存传递的缺陷列表对象

## 输出
落盘文件:`skill/mcp/output/{项目}-bugstats-{日期}.json`
```json
{
  "total": 30,
  "byLevel": { "一级": 0, "二级": 3, "三级": 23, "四级": 4 },
  "byStatus": {
    "未关闭": 23,
    "已修复待回归": 7,
    "已延期": 1
  },
  "回归不通过": 4,
  "未关闭列表": [
    { "id": "1234", "级别": "三级", "模块": "电子围栏", "标题": "【电子围栏】进入围栏后未触发报警", "状态": "回归不通过" }
  ],
  "待回归列表": [
    { "id": "1235", "级别": "二级", "模块": "账号管理", "标题": "【账号管理】登录后Token过期", "状态": "已解决" }
  ],
  "byModule": {
    "电子围栏": { "未关闭": 5, "已修复": 2, "延期": 0, "回归不通过": 1 },
    "订阅与通知推送": { "未关闭": 10, "已修复": 0, "延期": 0, "回归不通过": 2 }
  }
}
```

## 统计口径(必须严格遵循)

| 字段 | 计算规则 |
| --- | --- |
| `total` | 输入文件中所有状态(激活/已解决/已延期/已关闭/…)的缺陷总数 |
| `byLevel` | 按缺陷原始 priority 字段聚合(一级/二级/三级/四级) |
| `byStatus.未关闭` | 状态为"激活-待确认"或"激活-已确认(回归不通过)"的缺陷，**不含已延期** |
| `byStatus.已修复待回归` | 状态为"已解决"的缺陷 |
| `byStatus.已延期` | 状态为"已延期"的缺陷，单独一列 |
| `回归不通过` | 状态为"激活-已确认(回归不通过)"的缺陷数 |
| `未关闭列表` | 状态为"激活-待确认"或"激活-已确认"的缺陷(用于第三节列表展示)，不含已延期 |
| `待回归列表` | 状态为"已解决"的缺陷(用于第四节独立列表) |
| `byModule` | 按标题【模块】前缀聚合(**见模块别名映射表**) |

## ⚠️ 模块别名映射(处理禅道前缀与测试方案命名不一致)
禅道缺陷标题的前缀与测试方案 1.4.1 全景表模块名**不完全一致**,采用确定性别名映射(每个前缀仅归一到唯一规范模块):

| 禅道实际前缀(归一化) | 测试方案模块 | 说明 |
| --- | --- | --- |
| 账号登录, 账号密码登录 | 🔐 登录与认证 | 登录认证 |
| 账号管理, 一级账号, 用户管理, 用户列表 | 👥 用户与账号体系 | 用户/账号管理 |
| 设备管理, 设备列表, 设备详情, 设备入库 | 📦 设备与分组管理 | 设备相关 |
| 普通通信, 消息管理 | 💬 消息通信 | 消息相关 |
| 救援棒报警 | 🚨 SOS 报警与报平安 | SOS报警 |
| 求救群聊, iOS求救群聊, 普通群聊 | 🆘 求救群聊 | 求救群聊 |
| 套餐商城 | 💰 套餐与扣费 | 套餐相关 |
| 电子围栏, 围栏报警记录 | 🗺️ 电子围栏 | 围栏相关 |
| WebSocket, 实时推送相关前缀 | 📡 WebSocket 实时推送 | 推送相关 |
| 消息通知, 公众号通知, 设备订阅, 通知记录 | 🔔 订阅与通知推送 | 通知相关(含消息通知) |
| 控制台, 大屏监控 | 监控平台 | 其他/控制台 |

**匹配规则(确定性,按顺序执行)**:
1. 取缺陷标题的【模块】前缀
2. 精确匹配别名表 → 唯一规范模块
3. 无精确命中 → 最长别名匹配 → 唯一规范模块
4. 仍未命中 → 归入"未分类"模块,**不静默丢弃**,在 `byModule` 中以 `未分类` 键记录,并输出告警
5. 一个缺陷只归入一个模块,禁止同时计入多个模块

## 关键约束(禁止违反)
1. **只算一次**:`bugStats` 一旦生成,本发布周期内只读不再重算
2. **禁止重计数**:钉钉报告和 Notion 报告中的每一个数字都必须从 `bugStats` 取值
3. **标题原样**:`未关闭列表` 和 `待回归列表` 中的标题取自禅道 MD 原文,**不许改写**
4. **模块别名必须应用(一对一)**:所有按模块分组的统计(`byModule`)必须经映射表归一化;每个前缀只映射到唯一规范模块,禁止一对多

## 执行步骤
1. 读取禅道汇总 JSON 文件(`{项目}-bugs-{日期}.json`)
2. 解析每条缺陷的 `title` / `severity` / `status` 字段
3. 按上述口径聚合生成 `bugStats` 对象
4. 落盘到 `skill/mcp/output/{项目}-bugstats-{日期}.json`
5. 输出统计摘要供 Agent 确认数字
