# 文档盘点技能 (doc-inventory)

## 一句话定位
给一个文件夹路径(可选关键词、可选时间筛选),自动扫描并统计该文件夹下所有文档:
**按文件格式分组计数** + **逐个罗列文件名/路径/创建时间/修改时间/大小/命中关键词**,
最终导出一份 Markdown 报告到被搜索的文件夹下。

> 这是「统计」类技能,第一性原则是**结果可重复(幂等)**:同一查询跑两次,结果必须一致。
> 本技能最大的敌人是"自我污染"——上一次生成的盘点报告会被当成文档再统计一次。所以默认排除自己生成的报告文件名。

---

## 输入契约(运行时向用户确认,全部可配)

| 参数 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| 文件夹路径 | ✅ | — | 要盘点的目录,绝对/相对路径均可,支持中文路径 |
| 关键词 | ❌ | 空=全量 | 可选。不填=盘点该文件夹**全部**文档;填了=只保留文件名含关键词的。支持多关键词逗号分隔(OR),每个文件标注命中了哪个词 |
| 时间筛选 | ❌ | 不限 | 可选。支持相对(`今日`/`本周`/`本月`/`今年`/`最近30天`)与绝对区间(`2026-07-14~2026-07-18` 或 `2026/07/14-2026/07/18`) |
| 时间字段 | ❌ | 修改时间 | 可选。按"创建时间"筛还是"修改时间"筛,默认修改时间 |

**问参数的姿势**:不要一次性甩一堆问题。路径必问;其余三项用一个合并的问法:"需要按关键词或时间过滤吗?(不需要就盘点全部)"。

---

## 关键词匹配规则(钉死,避免歧义)

| 维度 | 规则 | 为什么 |
| --- | --- | --- |
| 匹配范围 | **仅文件名**(扩展名前的部分),不含正文 | 本技能是"找文件",不是"搜内容";搜内容属于另一个技能 |
| 大小写 | 忽略大小写(英文统一转小写比对) | 文档命名不规范是常态,`Test/TEST/test` 应视为同 |
| 匹配方式 | 子串包含 | `测试` 命中 `API测试框架`、`测试报告`;`test` 命中 `latest` |
| 多关键词 | 逗号分隔=OR(命中任一即保留) | 每个文件在报告里标注实际命中了哪个词(`matchedBy`) |
| 英文短词警示 | 当关键词是 `test/go/id` 等 ≤4 字母英文词,报告里提示"可能误命中(如 latest/contest)" | 子串包含对短英文词假阳性高,提醒用户即可,不强行过滤 |

## 时间口径策略(自动判断,用户无需操心)

不同目录性质用不同时间源,技能自动检测并**在报告里显式标注用了哪个口径**(绝不撒谎):

| 目标目录性质 | 时间口径 | 备注 |
| --- | --- | --- |
| 在 git 仓库内 | 优先 git 提交时间(首次提交≈创建、最近提交≈修改),文件系统时间作补充 | git 时间最可信 |
| **普通文件夹(如 D 盘/桌面,非 git)** | 直接用文件系统时间 `CreationTime`/`LastWriteTime` | 对用户自管的上传目录,文件系统时间就是事实源,可信 |
| 无法判断 | 文件系统时间,标注"非 git 托管" | 兜底 |

> 实话:文件系统的"创建时间"在拷贝/同步时会重置,但对**用户自己上传整理的 D 盘文件夹**而言它仍是可信事实源。技能会如实标注口径,让用户自己判断是否采信。

## 默认排除集(内置,报告里显式列出)

这些目录/文件默认不统计,因为它们会污染结果:

**目录类**:`node_modules`、`.git`、`output`、`dist`、`build`、`__pycache__`、`.venv`、`venv`、`target`、`.idea`、`.vscode`、`bin`、`obj`

**文件类**:
- 自身产物:`文档盘点报告-*.md`(防自我污染,保幂等)
- Office 临时文件:`~$` 开头(打开中的 Word/Excel 临时锁文件)
- 系统垃圾:`Thumbs.db`、`desktop.ini`、`.DS_Store`、`*.tmp`

> 在报告的"总览"区显式列出本次排除了哪些规则,让结果可审计。用户如需放宽,运行时说明即可。

---

## 统计口径(必须严格遵循)

| 字段 | 计算规则 |
| --- | --- |
| 按格式分组 | 取文件**扩展名**,统一转小写归一(`.MD/.MARKDOWN→md`、`.DOCX/.DOC→docx系`)后计数,带数量与占比 |
| 无后缀文件 | 单独归"无后缀"类,**不丢弃、不报错** |
| 命中文档数 | 经过排除规则 + 关键词过滤 + 时间筛选后,剩余文件数 |
| 创建时间/修改时间 | 按时间口径策略取值,标注口径 |
| 大小 | 友好显示(B/KB/MB),保留1位小数 |
| 异常文件 | 0字节文件、>50MB超大文件、因长路径跳过的文件,单列"异常警示区" |

## 文档类型分类(测试资料分类统计)

这是本技能的进阶能力:在"按格式分组"之外,额外按**测试资料的业务类型**做分类统计,回答"测试报告多少份、说明书多少份、方案多少份、用例多少份"。规则可配置,放在 `classification-rules.json`。

### 默认分类规则

| 类型 | 匹配规则 | 优先级 |
| --- | --- | --- |
| 测试报告 | 文件名含「报告」 | 10 |
| 使用说明书 | 文件名含「使用说明 / 说明书 / 用户手册 / 操作手册 / 使用手册」任一 | 10 |
| 方案/计划 | 文件名含「方案」**或**「计划」 | 8 |
| 用例 | 扩展名为 `.xmind` 或 `.mm`(XMind/思维导图) | 5 |
| 其他/未分类 | 不命中以上任何 | — |

### 归类策略(必须遵循)

- **互斥归类**:一个文件只归一类。按 priority 从大到小逐条匹配,命中即归类、不再往下。这样**各类加起来=总数**,统计守恒。
- **优先级含义**:关键词类(报告/说明书/方案)优先级高于格式类(用例)。意思是:文件名有明确业务词时归业务类(如 `测试方案.xmind` 归"方案/计划"),纯脑图才归"用例"。这是为避免用例把带业务词的脑图吃掉。
- **候选标注不丢信息**:即便只归一类,每条记录会列出它还候选命中了哪些类(`candidates`),报告"按类型分类"节可附候选清单供核对。
- **用例的特殊性**:用例不靠关键词、直接按 `.xmind/.mm` 格式统计(主人指定)。脑图基本就是测试用例,但优先级低于关键词类——若某个 xmind 文件名带"报告/方案",优先归业务类。

### 可配置(改词加类不改脚本)

`classification-rules.json` 里改 `rules` 数组即可:改关键词、加新类型(如"需求文档""验收单""会议纪要")、调优先级。重新跑脚本即生效。脚本默认读取技能目录下的 `classification-rules.json`,也可用 `-RulesFile` 指定其他路径。

> 分类逻辑是对扫描结果的后处理,**与关键词/时间筛选叠加**:既能在全库上做分类统计(全库资料归到各测试类型),也能在"含某关键词"子集上做(先过滤再分类)。两个视角都支持,不破坏原有功能。

## 输出报告结构(严格用此模板)

> 必须把**全部命中文件的文件名 + 创建时间 + 修改时间**完整罗列进去,不能让用户去翻 JSON。
> JSON 只是程序二次处理用的副产物,不是给人看的载体。

## 执行流程(技能驱动 Claude 时的步骤)

1. **确认输入**:向用户拿到文件夹路径;关键词/时间筛选用一句话合并确认(可选)。
2. **跑兜底脚本**:调用 `scripts/scan_docs.ps1`(见下),产出 JSON。脚本负责扫盘、排除、时间取值、中文路径、长路径容错这些"PowerShell 一行写不清的活"。
3. **读 JSON,按报告模板生成 Markdown**:Claude 负责"把 JSON 翻译成人看的报告"(分组统计、占比条、排序、异常警示判断)。
4. **导出报告**:写到被搜索的文件夹下,文件名带时间戳。同时保留 JSON 副本。
5. **回报用户**:告诉用户报告路径、核心数字(命中文档数、按格式分组),并提示有没有异常警示。

> 为什么脚本只产 JSON、报告由 Claude 写?因为"占比条怎么画""异常怎么排序""警示文案怎么写"这些是呈现层,用模型做更灵活;而"扫几千个文件取准时间"这种确定性活,必须用脚本,不能让模型一个个 Read 文件(又慢又可能漏)。

## 报告生成硬约束(钉死,不得自由发挥)

历史的教训:旧版报告曾出现"明细只列 15 条就截断""分类统计整节缺失""目录分布用 ~380 这种约数"——全是模型生成报告时偷懒的结果。以下规则**不留模糊空间**,违反即视为生成错误,必须重做。

| # | 约束 | 具体要求 |
| --- | --- | --- |
| 1 | **全量原则** | 报告第二节必须罗列**全部**命中文件的文件名 + 创建时间 + 修改时间。**禁止任何"仅展示前 N 条""...其余见 JSON"式的截断。** 取消旧版">500 只出前 50 条"的上限。 |
| 2 | **分节必做** | 当 `summary.categoryEnabled=true`(分类规则加载成功)时,"按文档类型分类"统计表 + 第二节"按类型分节明细"**必须完整生成**,不得因"文件多"省略。即便没有分类规则,第二节也必须按**格式**分节全量罗列,不允许只给统计不给明细。 |
| 3 | **统计守恒** | 目录分布、按格式分组、按类型分组三处的数字必须是**精确计数**,**严禁 "~""约""左右" 等约数**(统计类技能用约数是硬伤)。生成后必须自检:`按格式分组总数 == 按类型分组总数 == summary.total`,三者不等即生成错误,重做。 |
| 4 | **大节二次分组** | 任一类型的文件数 > 50 时,该节内部**按格式再分小节**(表头 `#### · .xsd(N 个)`),每个小节内全量罗列。常见触发场景:目录混入大量 `.xsd/.py/.ttf` 代码/素材文件,全归"其他/未分类"导致该节膨胀。 |
| 5 | **文件名必完整** | 明细表"文件名"列必须显示**完整原文**,不可截断、不可省略扩展名。这是主人核心诉求,违反=功能失效。"完整路径"列可缩略,文件名不可。 |
| 6 | **时间口径不撒谎** | 报告头"时间口径"必须与 JSON 的 `summary.timeSource` 一致(git / filesystem),不得美化。明细时间值取自 JSON 的 `created`/`modified`,显示为 `YYYY-MM-DD HH:mm`。 |

> 自检口诀:**文件名一个不漏、分类总数等于命中文档数、统计数字不准约数、大节必拆小节。**

## 兜底脚本说明

`scripts/scan_docs.ps1` 是 PowerShell 脚本,参数:
- `-Path`(必填):目标文件夹
- `-Keyword`(可选):关键词,逗号分隔多词
- `-TimeFilter`(可选):`今日`/`本周`/`本月`/`今年`/`最近30天` 或 `2026-07-14~2026-07-18`
- `-TimeField`(可选,默认 `Modified`):`Created`/`Modified`
- `-OutFile`(可选):JSON 输出路径,默认输出到 stdout

脚本输出 JSON 结构见 `scan_docs.ps1` 头部注释。脚本内置默认排除集与自我污染排除。

> 脚本存在的理由:文件系统时间读取、中文路径、Windows 长路径(>260)容错、git 时间检测这些活,PowerShell 一次成型;让模型逐文件 Read 既慢又会漏大文件目录。脚本**只产数据**,不产报告——报告由模型按模板写。

## 边界与保护

- **全量不设上限**:**不再有"命中 >500 只出前 50 条"的旧规则**(已废除)。无论命中文件多少,报告第二节都按类型/格式分节全量罗列。大节触发"二次分组"规则(见硬约束第 4 条)保住可读性,而非靠截断省行。
- **报告体积是预期行为**:盘上千文件时,生成的 Markdown 可能有几千行,这是**正确**的——"完整罗列"本来就是主人的诉求。模型只需把 JSON 完整写入文件、按模板全量渲染报告即可,不必担心 md 体积或撑爆上下文(报告是写到磁盘文件,不是塞进对话上下文)。
- **可重复性**:排除自身产物 `文档盘点报告-*`,保证同目录连跑两次结果一致。
- **路径过深**:捕获异常,计入"跳过文件",不中断扫描。
- **中文编码**:脚本统一 UTF-8 输出,中文文件名/路径不乱码。
