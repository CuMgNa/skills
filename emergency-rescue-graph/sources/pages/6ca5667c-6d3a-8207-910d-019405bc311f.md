## 求救群聊整体规划文档（小程序侧聚焦）
### 1. 目标与范围
- **目标**：沉淀一份围绕天通应急救援棒的求救群聊机制规划，作为后续 PRD 修订、研发设计与测试用例设计的统一依据。
- **范围**：仅针对 **天通应急救援棒** 报警触发的求救模式，涵盖：
	- 求救群聊的触发机制（何时创建 / 何时不创建）。
	- 自动拉取群成员规则（含标签、层级、紧急联系人、解绑/冻结等边界）。
	- 群聊状态流转与消息发送控制（救援中 / 救援完成）。
	- 小程序端与 Web 端在逻辑上的一致性（交互差异不在本计划展开）。
---
### 2. 求救群聊触发机制规划
- **设备类型前置校验**
	- 仅当设备 `deviceType = 天通应急救援棒` 时，才进入求救模式逻辑。
	- 其他设备类型触发报警时，走监控平台旧流程，不创建求救群聊。
- **触发入口**
	- 终端侧：天通应急救援棒**手动 SOS** 或 **自动报警**（如落水、拆卸等）上报报警消息。
	- 平台侧：US-010 所述”应急救援棒触发报警机制”，经接入层 → 业务平台 → 通信模块。
- **是否创建求救群聊的判定**
	- **报警瞬间存在有效绑定关系**（至少存在一个业务账号或紧急联系人）：
		- 若该设备当前**无”救援中”群聊** → 在 10 秒内创建新的求救群聊，群ID 格式 `SOS-设备ID-群编号`，状态置为 **救援中**。
		- 若已存在”救援中”群聊 → 不重复创建，仅更新群内设备状态 / 报警信息（逻辑上仍保证同一设备只存在一个救援中的群）。
	- **报警前已取消绑定，且无任何紧急联系人**：
		- 不创建新的求救群聊，仅记录报警日志与设备状态（特殊故障场景，不对外暴露协同群）。
	- **群聊创建失败的容错处理**：
		- 在满足上述创建条件时，如因异常导致创建求救群聊失败，本版本**不做额外自动重试/补偿**；
		- 等待设备后续再次触发报警时，再按本节规则重新尝试创建新的求救群聊。
- **报警后的解绑影响**
	- 详见 3.5「只增不减原则」：群聊创建并拉完成员后，后续解绑/标签变更不自动改动原有群成员列表，确保事件闭环数据完整。

### 3. 自动拉取群成员规则规划
### 3.1 成员类型定义
- **个人主账号 A**：
	- 在绑定关系中，标签为 `我的` 的个人账号。
	- 若存在，视为该终端的拥有者 / 购买者，**必拉**。
- **企业账号（最低等级 E）**：
	- 所有对该设备绑定且设备标签为 `我的` 或 `用户` 的企业账号（一级 / 二级 / 三级）。
	- 从中选层级**最低**的企业账号作为 `E`，**必拉**（若存在）。
- **紧急联系人集合 C**：
	- 由个人主账号在小程序绑定设备时配置，数量 `0–3` 个。
	- 每个紧急联系人对应一个 **个人账号**（无则创建账号），全部 **必拉**。
	- 配置层面不允许重复手机号/联系人，因此运行时无需考虑”同一账号被配置多次”的场景。
- **设备虚拟成员 D**：
	- 天通应急救援棒本身作为虚拟成员，用于展示在线 / 离线 / 报警状态与消息路由。
- **账号状态**：
	- 账号为”正常”或”冻结”时，**都参与拉群**。冻结仅影响用户能否登录前端，不影响入群与消息/通知派发。
### 3.4 账号角色与结束求救权限（补充说明）
- **个人主账号 A**：
	- 具备”结束本次求救”的权限，可在群管理中点击”救援完成 / 救援已完成”按钮。
- **最低等级企业账号 E / 企业管理员**：
	- 具备与个人主账号等价的结束权限，可在群管理中点击”救援完成”按钮。
- **紧急联系人 C**：
	- 仅参与协同沟通与接收报警通知，**不具备结束求救的权限**，在群管理中不展示”救援完成”按钮或按钮置灰不可点。
- **系统客服 / 其他辅助角色**：
	- 默认不具备结束本次求救的权限，仅在特殊运营流程中通过后台工具处理，与小程序/Web 前端逻辑解耦。
### 3.2 拉群算法（口语版伪代码）
1. 报警瞬间，查询设备绑定关系与紧急联系人配置。
2. 若无任何业务账号 / 紧急联系人（含个人、企业、紧急联系人）→ 不创建求救群聊。
3. 若存在：
	- 若有个人主账号（标签为 `我的`）→ 拉入 A。
	- 在所有设备标签为 `我的` 或 `用户` 的企业账号中，按层级选出最低等级企业账号 E → 若存在则拉入。
	- 遍历紧急联系人列表：
		- 查找对应的个人账号；若不存在则创建后拉入。
		- 仅在”紧急联系人账号 ID 与 A 完全相同”这种极端情况下去重，其余情况一律保留（不会因为手机号相同而去重）。
	- 添加设备虚拟成员 D。
4. 将上述成员集合作为该次报警的「初始群成员列表」，写入求救群聊记录，用于后续小程序 / Web / 对讲机的入群展示。
### 3.3 已明确 / 不再展开的场景
- 紧急联系人数量超过 3 个、重复配置等，被前端与接口层约束为”**不可保存**”，运行态不再单测。
- 并发报警（极短时间多次 SOS）按”同一设备只允许一个救援中的求救群”处理，但不重点做高并发场景的功能测试。
- 系统客服暂不作为自动拉群成员，后续如需加入，单独设计”系统客服角色入群规则”。
### 3.5 报警期间账号变更与增量拉群规则
### 3.5.1 首批成员快照（报警瞬间）
- 第一次报警创建求救群时，以**报警瞬间**的绑定关系和紧急联系人为准，按 3.2 拉群算法计算出首批成员并拉入群中。
- 首批成员一经确定，后续绑定关系变更（新增绑定 / 解绑 / 标签调整）**不自动改动**当前群成员列表。
### 3.5.2 增量拉群（同一场报警内再次上报）
当设备仍处于”报警中”并再次上报报警时（复用现有”救援中”群，不创建新群），执行以下增量拉群流程：

### 3.5.3 只增不减原则
在同一场报警期间，自动算法**只做”增量拉入”，不做”自动剔除”**：
- 即使某些账号中途解绑设备、或标签从”我的/用户”变为”好友”等，系统也**不会**自动将其移除出求救群。
- 目前求救群聊没有退群操作入口
- 目的：避免责任链断裂，保证历史消息和沟通链路的连续性。
> **注意：只增不减仅针对”群成员列表”维度。** 个人主账号 A（绑定关系为「我的」）解绑设备虽然**不影响群成员列表**（A 及所有成员仍在群内），但会**触发求救群聊状态自动变更为”救援完成”**（详见 §4.1），群聊进入只读态，全端禁发消息。企业账号或其他成员解绑设备则**既不影响成员列表，也不影响群聊状态**。

### 4. 群聊状态流转与消息控制规划
### 4.1 群聊状态
- **救援中**：
	- 初始状态：设备报警创建群聊，或已存在群聊被再次报警激活为”报警救援”。
	- 行为：
		- 所有成员可在群内发送文本 / 语音消息。
		- 消息按 US-007 规则转发至终端 / Web / 小程序 / 对讲机，并按 COM-R11 更新”等待发送→发送中→终端未读→终端已读”等状态。
- **救援完成**：
	- 触发条件（任一满足，**仅针对天通应急救援棒**）：
		- 设备在终端侧主动上报”取消 SOS”。
		- 群内最后一条消息时间起超过 72 小时无新消息/短音（超时自动关闭本次求救）。
		- 个人主账号 / 最低等级企业账号在群管理中点击”救援完成 / 救援已完成”。
		- **个人主账号与天通救援棒的绑定关系为「我的」，且在设备管理中解绑该设备** → 求救群聊状态自动变为”救援完成”（等同于手动结束）。
	- 一旦进入”救援完成”，需**同时**对天通应急救援棒执行：
		- 将设备报警状态从”报警中”标记为”已解除”，并将设备通信模式从求救模式恢复为普通模式。
		- 将当前求救群聊状态从 `rescuing` 切换为 `completed`，后续仅保留为历史记录。
		- **取消该群会话中处于”等待发送给终端”状态的所有消息并按规则退费**（详见 `sos-msg-send-limits` §4.3）。
	- 行为（群聊侧）：
		- 所有人 **不可再发送文本/语音**，前端输入框置灰或隐藏，并提示”该群聊已救援完成，无法发送”；后端拒绝发送请求。
		- 仍可进入群聊查看历史消息、套餐信息与群管理信息（只读）。
	- 72 小时自动结束计时细节：
		- 仅统计业务消息：**系统消息（如成员入群、状态变更提示等）不参与计时重置**；
		- 每当终端上报业务消息（语音 / 报警位置）或群成员在会话中发送文本 / 语音时，均将”最后一条业务消息时间”更新为该条消息时间，并据此重新累加 72 小时。
### 4.2 状态切换关系
```mermaid
flowchart TD
    Created[新建求救群聊\n状态: 救援中] -->|设备持续报警/正常通信| InRescue[救援中]
    InRescue -->|设备取消SOS| Completed[救援完成]
    InRescue -->|3天无新消息/短音| Completed
    InRescue -->|主账号/企业点击\"救援完成\"| Completed
    InRescue -->|个人主账号解绑\"我的\"天通棒| Completed
    Completed -->|仅查看历史记录| Readonly[只读群聊]
```
- **救援中 → 救援完成** 时，需同步：
	- 群状态字段从 `rescuing` 改为 `completed`。
	- 禁用所有发送入口（Web / 小程序 / 对讲机），仅保留历史查看。
	- 对天通应急救援棒：同步更新设备报警状态为”已解除”、通信模式恢复普通模式，并在群头部 / 设备卡片中展示”最新一条消息 / 救援完成标识、最后通话时间”等信息。
	- 取消该群会话中处于”等待发送给终端”状态的所有消息并按规则退费（详见 `sos-msg-send-limits` §4.3）。
- **多触发方并发结束的处理（补充说明）**：
	- 群聊状态是全局的：一旦任意具备结束权限的主体（设备取消SOS、个人主账号、最低等级企业账号、**个人主账号解绑「我的」天通棒**）首先将状态切换为”救援完成”，后续同类结束请求仅视为对”已结束”状态的确认，不再改变状态。
	- 前端在接收到”已结束”状态后，应立刻同步刷新所有成员界面：列表状态改为”已结束”、聊天页输入框置灰 / 隐藏、“救援完成”按钮消失或置灰。
	- 若其他有权限成员在此时点击”救援完成”，后端应返回”已结束 / 幂等成功”类结果，前端给出轻量提示（例如”该求救已由其他成员标记为救援完成，无需重复操作”），并以最新状态为准刷新界面。
	- 建议在群聊时间线中追加一条系统消息记录结束行为，例如：“【系统消息】2026-03-13 10:21，张三（企业账号）将本次救援标记为已完成”，用于责任追溯。

### 4.3 设备在线/离线与报警状态的关系
### 4.3.1 两个独立状态维度
设备同时维护两个**互相独立**的状态字段：
<table>
<tr>
<td>状态维度</td>
<td>可选值</td>
<td>判定方式</td>
<td>变更时机</td>
</tr>
<tr>
<td>在线状态</td>
<td>在线 / 离线</td>
<td>心跳上报：连续 1 分钟无心跳 → 判定离线；收到心跳或 TCP 重连 → 恢复在线</td>
<td>实时、自动</td>
</tr>
<tr>
<td>报警状态</td>
<td>正常 / 报警中 / 已解除</td>
<td>终端 SOS/落水等触发 → 报警中；4.3.3 中四种路径之一触发 → 已解除</td>
<td>事件驱动</td>
</tr>
</table>
> 关键约束：**在线状态变更不影响报警状态，报警状态变更也不改写在线状态**。二者各自独立更新。
### 4.3.2 四种组合状态与图标展示（仅针对天通应急救援棒）
以”**主体图标颜色 + 右上角角标**”组合表达，**图标状态实时更新**（心跳/报警状态变化时立即刷新前端展示）：
**① 报警中 + 在线**<br>- 主体图标：红色（表示设备当前处于报警中）<br>- 右上角角标：绿色 ✓（表示当前在线）<br>- 求救群聊状态：救援中，正常收发消息
**② 报警中 + 离线**<br>- 主体图标：红色（报警优先，未解除前始终红色）<br>- 右上角角标：灰色 ×（表示已离线）<br>- 求救群聊状态：救援中，下行消息进入等待队列，待设备重新上线后补发
> **报警中：主体图标始终红色，通过右上角 ✓ / × 区分在线 vs 离线**
**③ 已解除 + 在线**<br>- 主体图标：绿色（恢复普通在线态）<br>- 右上角角标：**不展示**（角标位清空）
**④ 已解除 + 离线**<br>- 主体图标：灰色（普通离线态）<br>- 右上角角标：**不展示**（角标位清空）
> **报警一旦解除：红色主体 → 恢复绿/灰；右上角 ✓ / × 一律取消，在线/离线仅依赖主体颜色识别**
**设计逻辑总结：**<br>- 报警中 → 主体锁定红色，角标辅助区分在线/离线（因为红色已占用主体颜色位）<br>- 非报警 → 主体颜色直接表达在线（绿）/离线（灰），无需角标辅助
### 4.3.3 报警状态解除的四条唯一路径（仅天通应急救援棒）
报警状态从”报警中”变更为”已解除”，**有且仅有**以下四种触发方式（与第 4.1 节救援完成触发条件一致），不存在其他隐式清除：
1. **终端取消 SOS**：设备主动上报”取消 SOS”。
2. **有权限成员手动结束**：个人主账号 A / 最低等级企业账号 E 在群管理中点击”救援完成”。
3. **72 小时无新业务消息自动结束**：计时规则见第 4.1 节。
4. **个人主账号解绑「我的」天通棒**：个人主账号在设备管理中解绑与天通救援棒的「我的」绑定关系，触发三联动：① 设备报警状态→已解除；② 通信模式→恢复普通模式；③ 求救群状态→救援完成（只读）。
> 以上任一路径触发时，系统**同时**执行：<br>① 设备报警状态 → 已解除；② 通信模式 → 恢复普通模式；③ 求救群状态 → 救援完成（只读）。
### 4.3.4 图标实时更新机制
- **心跳恢复**（离线→在线）：角标从 × 实时切换为 ✓（报警中），或主体从灰色切换为绿色（非报警）
- **心跳超时**（在线→离线）：角标从 ✓ 实时切换为 ×（报警中），或主体从绿色切换为灰色（非报警）
- **报警解除**：主体从红色恢复为绿/灰，角标 ✓/× 立即清除
- **触发报警**：主体从绿/灰切换为红色，同时根据当前在线状态显示角标 ✓ 或 ×
- 前端通过 WebSocket / 长连接 / 轮询等方式接收状态变更事件，确保多端同步

- **反向联动（求救群结束 → 报警记录自动处理，仅天通应急救援棒）**：
	- 当本次求救群聊状态从”救援中”切换为”救援完成/只读”时，系统需将**对应的报警记录**自动标记为”已处理”，并写入”处理结果”（四选一）：
		- 终端取消SOS
		- 手动救援完成
		- 72小时自动结束
		- 个人主账号解绑设备
	- “处理结果”的取值规则：与触发本次”救援完成”的四条唯一路径一一对应（见 4.1 与 4.3.3）。
### 4.3.6 三重通知与报警状态的联动
- 同一场报警中，设备多次上报报警信息，**不重复**触发三重通知（ALM-R01/ALM-R02）。
- 仅当报警被正式解除（4.3.3 中任一路径）后，设备再次触发新报警，才会重新触发一轮三重通知。
### 4.3.7 设备类型差异（天通应急救援棒 vs 其他设备）
- **天通应急救援棒**：
	- 触发 SOS/落水报警时进入求救模式，自动创建求救群并按本规划拉群。
	- 报警状态的解除仅通过 4.3.3 中四种正式路径。
	- 报警结束时，系统同时将求救群置为”救援完成（只读）“、将设备报警状态置为”已解除”，并将通信模式恢复为普通模式。
	- 报警信息模块中对天通应急救援棒的”处理报警”操作**不改变**设备报警状态，也**不触发**求救群状态变化，仅影响该条报警记录的处理标记和后续三重通知控制。
- **其他设备（如北斗终端）**：
	- 触发报警时**不进入**本求救群机制，仍走监控平台既有报警处理流程。
	- 报警信息模块中的”处理报警”行为保持旧逻辑：可将报警状态标记为已处理/已解除，并驱动相应的告警关闭、通知终止等后续动作。

### 4.4 求救群聊命名规则（补充说明）
- **内部命名与唯一标识**：
	- 为便于后端与运维定位，同一设备的每一场独立求救会话，建议使用统一格式的内部名称/ID：
		- `SOS-<设备ID>-<四位序号>`，例如：`SOS-42535215-0001`、`SOS-42535215-0002`。
	- `<设备ID>` 为天通应急救援棒的唯一标识；`<四位序号>` 从 `0001` 开始递增，范围 `0001–9999`，表示该设备第几次触发求救会话。
- **与”同设备仅一个救援中群聊”规则的协调**：
	- 业务上仍遵守”同一设备在任一时刻仅存在 1 个状态为『救援中』的求救群聊”的约束：
		- 若上一场求救的群聊仍处于”救援中”，再次报警不创建新群，而是复用该群，仅在群内追加报警信息。
		- 若上一场求救已进入”救援完成”（只读）状态，再次报警时可创建新的一场会话，内部 ID 中的序号递增（例如从 `…-0001` 到 `…-0002`），以区分多次历史求救。
- **小程序/Web 前端展示名称（极简风格）**：
	- 为避免标题过长、影响列表可读性，前端展示可采用简化规则：
		- 列表与聊天窗口标题可显示为 `SOS-<设备ID>` 或 `SOS-<设备ID>-<四位序号>`，保持以设备ID为核心识别信息。
		- “救援中 / 已结束”等状态通过右侧状态标签或副标题展示，而非堆叠在群名中。
	- 设备昵称（如”小蓝”）可在群聊内设备信息栏或设备卡片中展示，不强制出现在群名称里，保持命名规则稳定、机器可读。
- **系统消息中的辅助说明**：
	- 若存在多次求救历史，建议在系统消息中补充文案，例如：“本次为设备 42535215 第 2 次求救”，用于帮助人工快速理解当前会话与历史的关系，而不强求在群名中承载所有信息。

### 5. 小程序端与 Web 端的一致性原则
- **核心业务逻辑一致**：
	- 求救群聊的触发、拉群成员、状态切换规则，以合并版 PRD（US-007/008/009/010）+ 20260105 总体文档为准，**Web 与小程序共用同一套后端逻辑**。
- **差异仅在交互与布局**：
	- Web：侧边栏设备列表 + 中央群聊窗口。
	- 小程序：底部 `通信消息` Tab + `普通通信/求救群聊` 标签页 + 单页聊天室。
	- 在计划与测试设计中，将交互差异单独归纳，不影响业务规则判断。
