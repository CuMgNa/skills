## 一、项目概述
在1期基础上，本期叠加**对讲群通信**、**星豆虚拟货币与套餐购买**、**设备入库与绑定**、**管理后台统一运营管理**四大模块，形成”通信 + 计费 + 设备 + 运营”完整闭环。本次评审范围覆盖三端：**小程序（tdwt-applet）**、**监控平台 Web（pg-podium-tdwt）**、**管理后台（monitor-admin）**。
**在范围内：** 对讲群全生命周期（创建/邀请/消息/套餐/结束）、星豆体系（充值/消费/套餐购买）、管理后台参数配置与统计、设备入库绑定（含跨企业解绑）、消息通知记录迁移。
**排除项：** 设备单聊通信（1期已有）、SOS基础触发逻辑（1期已有）、调度群聊功能（1期已有）、星豆提现/退款、监控平台其他无关模块。

## 四、模块一：对讲群
### US-群-01：创建对讲群
**描述：** 作为账号用户，我希望创建一个对讲群，以便将名下设备组织为一个通信单元，统一接收群内消息。
**前置条件：** 用户已登录；账号星豆余额充足（如管理后台已开启创建扣费）。
**触发方式：** 点击消息列表页「+」按钮 → 选择「创建对讲群」。
**主流程：**
```mermaid
flowchart TD
    Start["点击 + 创建对讲群"] --> InputName["输入群名称"]
    InputName --> CheckDup{"群名是否重复?"}
    CheckDup -->|"重复"| InputName
    CheckDup -->|"不重复"| ShowFee["展示创建费用"]
    ShowFee --> CheckBalance{"星豆余额充足?"}
    CheckBalance -->|"不足"| ShowRecharge["弹出充值引导弹窗"]
    CheckBalance -->|"充足"| Deduct["扣除星豆"]
    Deduct --> CreateResult{"创建成功?"}
    CreateResult -->|"失败"| Refund["自动退还星豆"]
    CreateResult -->|"成功"| Done["群出现在消息列表顶部，账号成为群主"]
```
**分支/异常：**
<table header-row="true">
<tr>
<td>场景</td>
<td>处理</td>
</tr>
<tr>
<td>群名重复（同账号下）</td>
<td>提示”群名已存在，请重新输入”</td>
</tr>
<tr>
<td>扣费成功但创建失败</td>
<td>自动退还星豆</td>
</tr>
<tr>
<td>星豆不足</td>
<td>弹出充值引导弹窗</td>
</tr>
</table>
**验收标准：**
- 群名长度 1\~15 字符，同账号下不可重名
- 创建成功后群出现在消息列表顶部，群主标识显示
- 扣费成功但创建失败时星豆自动退还，无需用户操作
- 成员上限按设备数计（个人5台/企业100台，均可配置）
**关键规则：**
- 创建扣费金额和开关均由管理后台配置；
- 企业子账号可独立创建群，成员上限与一级账号共用同一配置（REQ-001, REQ-028）。

### US-群-02：群信息查看
**描述：** 作为群成员，我希望查看群的基本信息和成员列表；作为群主，我还能看到管理入口。
**触发方式：** 在群聊页点击底部「群信息」按钮，或从消息列表长按进入。
**群主视角包含：** 群名称 / 成员列表（含头像、昵称、设备卡号）/ 套餐概览 Tab / 「结束群组」按钮 / 「编辑群名」按钮 / 「查看邀请记录」入口。
**普通成员视角包含：** 群名称 / 成员列表 / 套餐概览 Tab。无结束/编辑/邀请记录入口。
**验收标准：**
- 群主视角有结束、编辑、邀请记录三个入口；普通成员视角无上述入口
- 按钮级权限控制，无权操作时直接隐藏，不显示禁用态
**关键规则：** 无权操作时直接隐藏按钮，不返回错误提示（F-002, REQ-002, REQ-003）。
---
### US-群-03：结束群组
**描述：** 作为群主，我希望在不再需要该群时将其结束，群结束后不可恢复。
**触发方式：** 群信息页点击「结束群组」。
**主流程：**
```mermaid
flowchart TD
    Start["群主点击结束群组"] --> Confirm["二次确认弹窗"]
    Confirm -->|"取消"| End["结束，不操作"]
    Confirm -->|"确认结束"| EndGroup["群状态变为已结束"]
    EndGroup --> InvalidInvites["所有待处理邀请自动失效"]
    InvalidInvites --> CheckRefund{"退还开关开启?"}
    CheckRefund -->|"是"| Refund["退还邀请星豆"]
    CheckRefund -->|"否"| Final["结束"]
    Refund --> Final
    Final --> Notify["所有成员消息列表显示灰色已结束标签"]
```
**验收标准：**
- 结束前必须有二次确认弹窗
- 结束后群状态变为”已结束”，所有成员消息列表显示灰色”已结束”标签
- 待处理中的邀请自动失效，星豆退还（如退还开关开启）
- 三端统一使用”结束群组”文案
- 群主账号过期或冻结时，群组不做特殊处理
- 群主账号被删除时，系统自动结束该群（流程同本故事，待处理邀请按退还开关处理）
**关键规则：**
- 术语统一为”结束群组”（DEC-002, REQ-004）；
- 群主过期/冻结不触发群状态变更；群主被删除时自动结束群（TBD-001）。

### US-群-04：添加成员（邀请）
**描述：** 作为群主，我希望通过邀请设备加入群，扩大对讲范围；我还能查看所有已发出的邀请历史。
**前置条件：** 当前用户为群主；群内成员数未达上限。
**触发方式：** 群聊页底部「添加成员」（仅群主可见）或群信息页入口。
**设备类型限制：** 设备列表只显示\*\*天通应急救援棒（TT_RESCUE_STICK）\*\*类型的设备，其他类型不可加入对讲群。
**设备列表过滤规则：** 个人账号邀请时，仅展示已绑定个人账号的设备；未绑定个人账号的设备及企业账号绑定的设备不在列表中显示（企业账号绑定的设备只能由企业账号直接加入）。

**主流程：**
```mermaid
flowchart TD

    CheckFull -->|"已满"| ErrFull["提示：成员数已满，结束"]
    CheckFull -->|"未满"| SelectDevice["从天通救援棒设备列表选择设备（手动填卡号或扫码SN）"]
    SelectDevice --> CheckBalance{"星豆余额充足?"}
    CheckBalance -->|"不足"| ErrBalance["提示：扣费失败，星豆不足请先充值"]

    CheckOwner -->|"自己账号的设备"| CheckInGroup{"设备已在其他活跃群?"}

    ConfirmSwitch -->|"确认"| Deduct
    CheckOwner -->|"他人个人账号的设备"| DeductAndInvite["展示费用，扣除星豆"]
    DeductAndInvite --> SendInvite["发送邀请通知给对方个人账号，见 US-群-05"]
```

**验收标准：**
- 设备列表只显示天通应急救援棒类型设备
- **个人账号邀请时，设备列表仅展示已绑定个人账号的设备**
- 个人账号邀请时，企业账号绑定设备不显示
- 企业账号绑定的设备只能由企业账号直接加入
- 自己的设备已在其他群时，弹出换群确认弹窗，取消则不扣费
- 邀请他人设备时立即扣费；拒绝/超时按开关决定是否退还
- 移除成员和设备换群退出均**不退还**邀请星豆
- 成员昵称最大 15 字符
- 个人账号扫码发现设备未绑定个人账号，提示“设备未绑定个人账号”。
- 企业账号扫码发现设备非本账号所有，提示“未绑定该设备”。
**关键规则：**
- 邀请时立即扣费；
- 批量邀请时，如果因为各种原因导致一个失败，则全部失败
- 移除/换群退出不退还星豆，只有邀请失败/拒绝/超时才按开关退还（REQ-005, REQ-006, REQ-008, REQ-009）。

### US-群-05：接收与处理邀请
**描述：** 作为被邀请设备的账号拥有者，我希望收到入群邀请通知，并在限定时间内选择接受或拒绝。
> 本故事只适用于邀请他人设备的场景（路径2）；邀请自己设备不触发此流程。
**触发方式：** 设备真正主人收到小程序内邀请通知（不推送微信服务通知，见 TBD-002）。
**主流程：**
```mermaid
flowchart TD
    Receive["设备拥有者收到邀请通知"] --> ShowDialog["打开邀请确认弹窗（邀请方信息、群名称、费用说明、换群提醒、服务端倒计时）"]
    ShowDialog --> CheckResponse{"用户在倒计时内响应?"}
    CheckResponse -->|"超时未响应"| Expire["邀请自动失效"]
    Expire --> CheckRefund{"退还开关开启?"}
    CheckRefund -->|"是"| Refund["退还星豆"]
    CheckRefund -->|"否"| End["结束"]
    Refund --> End
    CheckResponse -->|"拒绝"| Reject["邀请失效，不退还星豆"]
    Reject --> End
    CheckResponse -->|"同意"| CheckCurrentGroup{"设备当前在其他活跃群?"}
    CheckCurrentGroup -->|"是"| ExitOld["自动退出原群"]
    CheckCurrentGroup -->|"否"| JoinNew["直接加入新群"]
```

### US-群-14：对讲群 PTT 语音广播（服务端下行转发）
**描述：** 作为系统，我希望在收到对讲群内某台设备的语音消息后，将该消息下行广播给群内所有其他终端设备，实现真正的 PTT（Push-To-Talk）对讲功能。
**触发条件：** 语音消息路由结果为”转发至对讲群”（见 US-群-11）。
**主流程：**
```mermaid
flowchart TD
    Receive["收到设备A的语音消息，路由目标为对讲群"] --> CheckSenderBalance{"发送方设备A套餐是否充足?"}
    CheckSenderBalance -->|"不足"| RecordFail["记录失败，停止下行转发"]
    CheckSenderBalance -->|"充足"| DeductSender["扣除发送方设备A 1条短音额度"]
    DeductSender --> QueryMembers["查询对讲群成员列表，排除发送方设备A"]
    QueryMembers --> ForEachReceiver["逐台接收设备处理"]
    ForEachReceiver --> CheckReceiverBalance{"接收设备套餐是否充足?"}
    CheckReceiverBalance -->|"不足"| MarkReceiverFail["该设备记为失败，不转发"]
    CheckReceiverBalance -->|"充足"| DeductReceiver["扣除该接收设备 1条短音额度"]
    DeductReceiver --> Forward["下行转发语音至该设备"]
    MarkReceiverFail --> NextReceiver["下一台接收设备"]
    Forward --> NextReceiver
    NextReceiver --> WaitACK["等待各设备回传已读回执 ACK"]
    WaitACK --> UpdateStatus["更新消息气泡已读/未读/失败状态"]
    RecordFail --> End["结束"]
```
**消息状态定义：**
- 未读：消息已下发但尚未收到接收端ACK回执。
- 已读：接收端已返回已读回执。
- 失败：下发成功但超时未收到ACK，或因余额不足导致无法转发。
- 新增中间状态“等待下发中”，用于标识消息排队待发送阶段。
**关键规则：**
- 发送方设备 A 发送时扣除 1 条短音；不收到自己的消息回传
- 发送方套餐余额不足时，**停止全部下行转发**并记录为失败
- 每台接收终端成功收到下行语音时，从该接收设备套餐扣除 1 条短音；接收方余额不足时，仅该设备记为失败，不影响其他设备
- 已读/未读/失败状态以终端 ACK 为准；对讲群 ACK **独立存储**，不复用 EmergencyMessageStatus
**验收标准：**
- 发送方套餐充足时，余额充足的接收设备均收到下行语音
- 每条对讲群语音消息：发送方扣 1 条短音，每台成功接收的终端各扣 1 条短音
- 消息气泡实时更新”X人已读 / Y人未读 / Z人失败”
- 发送方套餐余额为 0 时，停止全部下行转发；某接收方余额为 0 时，仅该设备失败

## 七、全局业务规则
以下规则跨模块生效，各用户故事中不再重复说明。
1. **星豆不可提现、不可退款**，仅限系统内消费
2. **一台设备同一时间只能处于一个未结束的对讲群**，接受新邀请时自动退出原群
3. **成员上限配置变更只对新建群生效**，不影响已有群
4. **账号冻结或过期后**，该账号无法在群内（通过终端）发消息，但保留在成员列表中（不自动退群）；**群主账号过期或冻结时，群组不做特殊处理**；**群主账号被删除时，系统自动结束该群**（流程同 US-群-03，待处理邀请按退还开关处理）
5. 三端统一使用 “**结束**” 作为结束群的术语
6. **星豆退还（如邀请失败/群结束）只退还一次**，防止重复退还
7. **退款统计口径**：只含套餐退款，星豆退还在星豆明细单独统计，不计入退款
8. **子账号星豆独立**，不与一级账号共享；充值入口对所有账号类型开放
9. **同一手机号可拥有企业账号和个人账号两个独立身份**（小程序切换登录）；对讲群、星豆、套餐均按身份隔离，两空间完全独立，不提供合并视图
10. **对讲群按账号严格隔离**：企业父账号与子账号的对讲群互不可见；每个账号只能管理自己创建或加入的群
11. **Web 端与小程序对讲群操作权限完全对齐**，不存在端级别的功能限制
12. **套餐绑定设备，不绑定账号**：无论谁购买、谁创建群，通信费用均从**各设备自身绑定套餐**中扣除，与账号身份无关
13. **对讲群 SOS 报位**：仅**带 SOS 标识**的位置消息在对讲群产生报位气泡并扣发送方报位时长；同一条 SOS 报位多目标路由时**发送设备只扣一次**；**普通报位不在对讲群聊产生报位气泡**
14. **对讲群 PTT 语音**：发送方发送时扣 1 条短音；下行广播至每台接收终端时，**各接收设备各扣 1 条短音**；发送方或接收方套餐余额不足时，停止对相应设备的下行转发并记录为失败，上行暂不限制
15. **对讲群成员只能是天通应急救援棒（TT_RESCUE_STICK）设备**，其他设备类型不可加入
16. 个人账号邀请只能邀请已绑定个人账号的设备，选择邀请设备时提前过滤掉未绑定个人账号的设备，企业账号绑定的设备只能由企业账号直接加入
17. **企业账号只能看到自己创建的群**
18. **移除成员和设备换群退出均不退还邀请星豆**；只有邀请失败/拒绝/超时才按退还开关处理
19. **群主结束群后仍可进入只读查看历史消息**；普通成员退出/被移除后不可查看历史
