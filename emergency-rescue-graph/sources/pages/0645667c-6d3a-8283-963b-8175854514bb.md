# 消息回执方案

<!-- notion_page_id: 0645667c-6d3a-8283-963b-8175854514bb -->

## 1. 术语与对象
### 1.1 群成员（求救群聊成员集合）
求救群聊成员，按 `sos-groupchat-plan_fa323615.plan.md` 的口径：
- **个人主账号 A**：绑定关系标签为 `我的` 的个人账号（存在则必拉）。
- **最低等级企业账号 E**：对设备绑定且标签为 `我的/用户` 的企业账号中，层级最低者（存在则必拉）。
- **紧急联系人集合 C**：主账号配置的 0–3 个紧急联系人，对应个人账号（无则创建），全部必拉。
- **设备虚拟成员 D**：设备虚拟账号，承接设备状态展示与“作为接收者”的统计。
- **系统客服/其他辅助角色**：如业务后续拉入群，则视为群成员参与展示；否则不参与。
补充确认：
- **群内成员只有加入，没有退出**。
- **冻结账号仍计入历史与新消息的接收范围**：冻结仅限制前端登录，不影响数据流转与统计。
- **成员状态**：除设备虚拟成员外，群成员不区分在线/离线/报警状态；仅设备虚拟成员展示设备状态。
### 1.2 系统消息 vs 普通消息
- **系统消息**：群聊事件日志（系统时间 + 操作事件）。
- **普通消息**：群聊内的文本/语音等业务消息。
## 2. 系统消息展示规则（事件日志）
### 2.1 系统消息应覆盖的事件
在求救群聊时间线中追加系统消息，至少包含：
- 成员加入群聊
- 成员被补拉入群
- 新一轮报警激活当前救援群
- 手动点击「救援完成」
- 设备报平安导致求救结束
- 72 小时无新消息自动结束
### 2.2 系统消息展示规范
- **格式**：`系统时间 + 操作事件`
- **可见性**：对所有当前群成员可见
- **统计**：不参与群内已读/未读统计
- **弹窗**：不进入“消息接收人列表”弹窗
示例（文案可按 UI 规范微调）：
- `【系统消息】2026-03-17 10:21，张三加入群聊`
- `【系统消息】2026-03-17 10:25，李四被补拉入群`
- `【系统消息】2026-03-17 10:30，设备再次报警，已激活当前救援群`
## 3. 历史消息可见性（展示层规则）
### 3.1 规则
- **后续加入/补拉入群的成员**：进入群聊后，**可查看该求救群聊的全部历史消息**。
- 该规则用于救援协同补齐上下文，与“已读未读统计口径”解耦（见第 5 节）。
## 4. 消息接收人列表（求救群聊专属）
### 4.1 适用范围与入口
- **仅求救群聊提供**“消息接收人列表”能力；普通通信不提供。
- 仅当消息气泡下方出现 `群内X人未读` / `群内已读` 标识时：
	- 用户点击该标识，打开“消息接收人列表”弹窗。
### 4.2 弹窗结构
- 标题：`消息接收人列表`
- Tab：
	- `未读(n)`
	- `已读(m)`
- 列表项：
	- 头像 + 昵称 + 账号ID
	- 若为**设备虚拟成员 D**，需额外展示设备状态（在线/离线/报警）与其“未读/已读”归类
### 4.3 接收者集合的统计口径（核心规则）
对任意一条普通消息 `msg`：
- **接收者集合** = `msg` 发送当刻的“当前群成员集合”中，**过滤掉发送者本人**后的集合。
	- 发送者本人不进入“未读/已读”任一 Tab，也不计入 `n/m`。
	- 设备虚拟成员作为接收者参与统计（当其不是发送者时）。
说明：
- “过滤发送者”是统一规则：无论发送者是普通成员还是设备虚拟成员，均过滤。
- “当前群成员集合”会在同一场救援中因“再次报警触发增量拉群”而扩大，但只影响**后续新消息**（见第 4.4 节）。
### 4.4 增量拉群对接收人列表的影响
在同一场“救援中”的求救群聊里：
- 若设备再次报警触发“补拉入群成员”（参考 `sos-groupchat-plan` 的增量拉群规则）：
	- 新成员会进入**后续新消息**的接收者集合与统计；
	- **不会回写**到其入群前的历史消息接收者集合/统计中。
## 5. 群内已读/未读统计规则
### 5.1 触发规则（进入即读）
- **打开群聊页**视为“进入会话”：
	- 该成员对其“接收者集合内”的未读消息，统一标记为已读。
- 若成员在群聊页停留期间收到新消息：
	- 新消息到达且在会话内可见时，按“已读”处理（与“进入即读”一致性）。
### 5.2 统计范围与排除项
- 统计对象仅覆盖**普通消息**；系统消息不参与统计。
- 统计对象按第 4.3 节“接收者集合口径”计算：
	- **发送者本人永不参与**该消息的群内已读/未读统计。
### 5.3 与历史可见性的关系（必须解耦）
- 后补入群成员可以查看全部历史消息（第 3 节），但：
	- **不参与其入群前历史消息的接收者集合与已读/未读统计**
	- 即：不会因其“看了历史”而回写历史消息的 `未读/已读人数` 与明细列表
### 5.4 展示规则
消息气泡下方显示聚合标识（示例）：
- `群内2人未读`
- `群内已读`
点击上述“群内”标识，打开“消息接收人列表”弹窗（第 4 节）。
## 6. 发送失败消息的处理
对“发送失败”的普通消息：
- 仍保留消息记录；
- 仍展示“群内已读/未读”聚合标识；
- 仍可打开“消息接收人列表”弹窗查看明细。
